## Known Limitations

| Limitation | Notes |
|---|---|
| **One character per SimuCo account at a time** | DragonRealms only allows one active character per account. Attempting to log in a second character on an account already in use returns an "Invalid login key" error from the SGE server (and may also kick the first session). Lichborne now guards against this at the Add Character form — refuses the login with a clear message *before* hitting the server, and also translates the server's "Invalid login key" error into the same hint as a defense-in-depth measure. To run multiple characters concurrently, use multiple SimuCo accounts. |
| **Two characters with the same name on different accounts share one `{Character}.yaml`** | Profile files are keyed by character name only, not account. If two of your accounts each have a character named "Sekmeht", they'll share one profile file. Pre-existing limitation from v0.5.x; multi-character makes it more reachable. Workaround: rename one of the characters in-game. |
| **Sessions are not persisted across app launches** | Closing Lichborne ends every active session. On next launch you'll need to log each character in again via the Add Character (+) form. By design — future polish item to remember last-active characters and offer a "reconnect all" on launch. |
| **Map view state is per-character and doesn't follow you between tabs** | Each character's current map zone, selected room, and viewport position are independent. Switching tabs shows whatever zone that character is in (or last opened), not a unified map. Intentional — characters explore independently and a shared map across tabs would be confusing in boxing scenarios. |
| **The Variable Inspector dropdown lists all character scopes from your Lich install** | The Lich Dashboard → Variables tab populates its scope dropdown from every `uservars` row in `lich.db3`, which includes any character you've ever connected via this Lich install (across all your accounts). Useful (you can inspect any character's vars from any tab) but worth knowing — your alt's vars are visible from your main's dashboard. **As of v0.9.0 the Variables tab is editable** (add / edit / delete) — but only for the *connected character's own scope*; viewing another character's scope stays read-only, because Lich variable writes can only safely target the session Lichborne is attached to. |
| **A character with a Lich Scripts panel open polls `;listall` every 5 seconds** | Per-character script tracking injects `;listall` every 5s — but **only while that character has a Lich Scripts panel open** (v0.13.1, B163); with the panel closed it injects nothing. Cheap server-side, and now scoped to characters actually watching scripts. |
| **Ruby GTK script windows — addressed in v0.9.1, pending verification** | Lich scripts that build their own OS-level window via `Gtk.queue { ... }` + `Gtk::Window.new` were flaky in earlier versions: **(a)** the window might not paint; **(b)** it might paint only **intermittently** — `kill-counter.lic` (see [Elanthipedia](https://elanthipedia.play.net/Lich_script_repository#kill-counter)) often surfaced only after a start→kill→restart cycle; **(c)** even once painted, interacting with a widget could crash Lich (the `;vars setup` shape, B138). **v0.9.1 reworked the Lich spawn** to fix the suspected cause: Lichborne now launches `rubyw.exe` as a normal Windows GUI-subsystem process (no `windowsHide`, no stderr pipe — stdout/stderr go to `{userData}/Logs/lich-launch/{Character}.log`) instead of a hidden console process. This is the community-standard shape (matches Frostbite/Genie) and gives Ruby/GTK the process/desktop context its windows need. The connect model itself was already correct (Lich force-mode; verified against `main.rb`/`argv_options.rb`) — only the spawn shape changed. **This REVERSES the earlier "declined to add GTK support" decision (2026-05-30/06-01)**: GTK is now expected-working, **pending empirical verification** (start `;vars setup` / `kill-counter` and confirm the window paints + survives interaction). If a GTK script still misbehaves on v0.9.1+, it's now a real bug worth investigating (capture the launch log), not expected behaviour. **The v0.9.0 proactive GTK advisory (the `client-notice` channel + the `.lic`-source GTK detector) was REMOVED in v0.9.1** — it assumed GTK is broken, which is no longer the operating assumption. Historical root-cause notes (still useful background): the GTK main loop WAS running the whole time ([`lich.rbw:135`](C:\temp\lich-dev\lich-5\lich.rbw#L135) calls `Gtk.main` unconditionally; [`gtk.rb:148-155`](C:\temp\lich-dev\lich-5\lib\common\gtk.rb#L148-L155) registers a `GLib::Idle` yield), so the failures were a subtler process-context problem — which the old console/hidden/piped spawn produced and the GUI-subsystem spawn removes. The `<streamWindow>` / `<pushStream>` XML approach (see `newkill-counter.lic`) remains the more portable option for script authors, and the in-app **Lich Dashboard → Variables** editor remains the recommended `;vars setup` replacement. CLAUDE.md "Lich launch & connect" + the GTK architecture note + pitfall #52 (and the #54 tombstone) document the launch shape and the reversal. |

### Death XML Reference (captured 2026-05-08)

Key XML sequences from a live death event, for use when implementing F10/F11:

**Death state indicators (on kill):**
```xml
<indicator id="IconKNEELING" visible="n"/>
<indicator id="IconPRONE" visible="y"/>
<indicator id="IconSTANDING" visible="n"/>
<indicator id='IconBLEEDING' visible='n'/>
<indicator id='IconDEAD' visible='y'/>
```

**Rank gain in exp component (F10):**
```xml
<component id='exp Crossbow'><b> Crossbow: 991 00% dabbling </b></component>
You've gained a new rank in using crossbows.
```

**Death's Sting rexp sequence (F11) — fires on EVERY exp update batch while active, not just once:**
```xml
<component id='exp rexp'>Rested EXP Stored: 3:56 hours Usable This Cycle: none Cycle Refreshes: 3:13 hours</component>
<component id='exp rexp'>[Because of Death's Sting, your rested exp is currently not being used.]</component>
```
The cycle refreshes timer counts down live (3:15 → 3:14 → 3:13 ...) in the first component.
When Death's Sting is no longer active, the second `exp rexp` component stops appearing.
Detection: if `skill === 'rexp'` and text starts with `[Because of Death's Sting` → set `deathsStingActive = true`; if `skill === 'rexp'` and text does NOT contain Death's Sting → clear it.

Partial reduction (alyssum tea, BOOST DEATH) produces plain text in main stream but does NOT change the rexp component — the Death's Sting message continues firing on every exp batch as long as any affliction remains:
```
You feel the weakness imparted by your recent death weaken, though some affliction still remains.
That was the last dose. You discard the remains of the alyssum tea.
```
**Full clearance confirmed.** The second `exp rexp` component simply stops appearing — no special clearance tag or message in the component stream. The plain text clear message appears in the main stream immediately before the next exp batch without it:
```
You feel the weakness imparted by your recent death slowly fade away.
```
Then the very next exp batch:
```xml
<component id='exp rexp'>Rested EXP Stored: 3:56 hours Usable This Cycle: none Cycle Refreshes: 3:06 hours</component>
<component id='exp tdp'> TDPs: 59864</component>
```
No second `exp rexp`. The `<pushBold/><popBold/>` pair that always trailed active batches is also absent after clearance.

**Final detection logic:**
- Active: `skill === 'rexp'` and text starts with `[Because of Death's Sting` → `deathsStingActive = true`
- Cleared: `skill === 'rexp'` and text does NOT start with `[Because` (i.e. normal rested exp summary) → `deathsStingActive = false`

**Death's Sting also appears in bold in the EXP command output (plain text, main stream, mono mode):**
```xml
<pushBold/>[Because of Death's Sting, your rested exp is currently not being used.]<popBold/>
```

**Body decay timer (plain text in main stream, sent once on death):**
```
Your body will decay beyond its ability to hold your soul in 250 minutes.
```

**Resurrection — indicators clear, spell/bleed reset, then room nav fires:**
```xml
<spell>None</spell>
<indicator id='IconBLEEDING' visible='n'/>
<indicator id='IconDEAD' visible='n'/>
... resurrection narrative text ...
<nav/>
<streamWindow id='main' title='Story' subtitle=" - [Temple of Light, Alcove of Eylhaar] (2012119)" .../>
```

**Vitals on resurrection (all bars reset to ~10%, recover over time):**
