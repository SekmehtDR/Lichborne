## Known Limitations

| Limitation | Notes |
|---|---|
| **One character per SimuCo account at a time** | DragonRealms only allows one active character per account. Attempting to log in a second character on an account already in use returns an "Invalid login key" error from the SGE server (and may also kick the first session). Lichborne now guards against this at the Add Character form ‚Äî refuses the login with a clear message *before* hitting the server, and also translates the server's "Invalid login key" error into the same hint as a defense-in-depth measure. To run multiple characters concurrently, use multiple SimuCo accounts. |
| **Two characters with the same name on different accounts share one `{Character}.yaml`** | Profile files are keyed by character name only, not account. If two of your accounts each have a character named "Sekmeht", they'll share one profile file. Pre-existing limitation from v0.5.x; multi-character makes it more reachable. Workaround: rename one of the characters in-game. |
| **Sessions are not persisted across app launches** | Closing Lichborne ends every active session. On next launch you'll need to log each character in again via the Add Character (+) form. By design ‚Äî future polish item to remember last-active characters and offer a "reconnect all" on launch. |
| **Map view state is per-character and doesn't follow you between tabs** | Each character's current map zone, selected room, and viewport position are independent. Switching tabs shows whatever zone that character is in (or last opened), not a unified map. Intentional ‚Äî characters explore independently and a shared map across tabs would be confusing in boxing scenarios. |
| **The Variable Inspector dropdown lists all character scopes from your Lich install** | The Lich Dashboard ‚Üí Variables tab populates its scope dropdown from every `uservars` row in `lich.db3`, which includes any character you've ever connected via this Lich install (across all your accounts). Useful (you can inspect any character's vars from any tab) but worth knowing ‚Äî your alt's vars are visible from your main's dashboard. |
| **Each connected character polls `;listall` every 5 seconds** | Per-character script tracking means each tab independently injects `;listall` into its Lich process every 5s. Cheap server-side, but scales linearly with character count. Future polish: opt out polling when the Active Scripts panel isn't visible. |
| **Ruby GTK script windows don't render** | Lich scripts that build their own OS-level window via `Gtk.queue { ... }` + `Gtk::Window.new` (the `kill-counter.lic` shape from the Lich script repository ‚Äî see [Elanthipedia](https://elanthipedia.play.net/Lich_script_repository#kill-counter)) silently fail to paint a window under Lichborne. The script's non-UI logic (counter increments, line filters, `_respond` calls) runs normally; only the GTK widget paint is missing. Root cause: Lichborne launches Lich with `--stormfront --dragonrealms` (force-mode), which skips Lich's entry GUI ‚Äî the entry GUI is what starts the GTK main loop, and `Gtk.queue` blocks need that loop to execute. The ruby-gtk3 gem still loads (so `HAVE_GTK` is `true` and the script's startup gate at line 5 doesn't trip), making the failure mode silent. Same behaviour Stormfront the official client exhibits when launching Lich the same way; not a Lichborne bug per se, just a property of the launch model. **Investigated and declined to fix** (2026-05-30, Sekmeht) ‚Äî Lichborne is positioned as a display/configuration layer over Lich, not a Lich-runtime replacement, and GTK widgets are a Lich-runtime affordance outside the front-end protocol Lichborne speaks. **Workaround for script authors**: use the `<streamWindow>` / `<pushStream>` XML protocol instead (see `newkill-counter.lic` for the minimal pattern: `_respond("<streamWindow id='X' title='X' />")` + `_respond("<pushStream id='X'/>content<popStream/>")`). The XML approach renders as a regular Lichborne panel and also works in every other front-end. CLAUDE.md (under "Lich launch & connect") documents the deflection path so a fresh-session investigation doesn't chase it as a bug. |

### Death XML Reference (captured 2026-05-08)

Key XML sequences from a live death event, for use when implementing F10/F11:

**Death state indicators (on kill):**
```xml
<indicator id="IconKNEELING" visible="n"/>
<indicator id="IconPRONE" visible="y"/>
<indicator id="IconSTANDING" visible="n"/>
<indicator id='IconBLEEDING' visible='n'/>
<indicator id='IconDEAD' visible='y'/>
```

**Rank gain in exp component (F10):**
```xml
<component id='exp Crossbow'><b> Crossbow: 991 00% dabbling </b></component>
You've gained a new rank in using crossbows.
```

**Death's Sting rexp sequence (F11) ‚Äî fires on EVERY exp update batch while active, not just once:**
```xml
<component id='exp rexp'>Rested EXP Stored: 3:56 hours Usable This Cycle: none Cycle Refreshes: 3:13 hours</component>
<component id='exp rexp'>[Because of Death's Sting, your rested exp is currently not being used.]</component>
```
The cycle refreshes timer counts down live (3:15 ‚Üí 3:14 ‚Üí 3:13 ...) in the first component.
When Death's Sting is no longer active, the second `exp rexp` component stops appearing.
Detection: if `skill === 'rexp'` and text starts with `[Because of Death's Sting` ‚Üí set `deathsStingActive = true`; if `skill === 'rexp'` and text does NOT contain Death's Sting ‚Üí clear it.

Partial reduction (alyssum tea, BOOST DEATH) produces plain text in main stream but does NOT change the rexp component ‚Äî the Death's Sting message continues firing on every exp batch as long as any affliction remains:
```
You feel the weakness imparted by your recent death weaken, though some affliction still remains.
That was the last dose. You discard the remains of the alyssum tea.
```
**Full clearance confirmed.** The second `exp rexp` component simply stops appearing ‚Äî no special clearance tag or message in the component stream. The plain text clear message appears in the main stream immediately before the next exp batch without it:
```
You feel the weakness imparted by your recent death slowly fade away.
```
Then the very next exp batch:
```xml
<component id='exp rexp'>Rested EXP Stored: 3:56 hours Usable This Cycle: none Cycle Refreshes: 3:06 hours</component>
<component id='exp tdp'> TDPs: 59864</component>
```
No second `exp rexp`. The `<pushBold/><popBold/>` pair that always trailed active batches is also absent after clearance.

**Final detection logic:**
- Active: `skill === 'rexp'` and text starts with `[Because of Death's Sting` ‚Üí `deathsStingActive = true`
- Cleared: `skill === 'rexp'` and text does NOT start with `[Because` (i.e. normal rested exp summary) ‚Üí `deathsStingActive = false`

**Death's Sting also appears in bold in the EXP command output (plain text, main stream, mono mode):**
```xml
<pushBold/>[Because of Death's Sting, your rested exp is currently not being used.]<popBold/>
```

Your body will decay beyond its ability to hold your soul in 250 minutes.
