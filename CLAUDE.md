# CLAUDE.md

AI-facing project guide. Loaded into every Claude session for this repo — keep tight, keep operational, keep current. **If you make a non-trivial architectural change, update this file.** The goal is that a fresh session can pick up the project without re-deriving its conventions from the code.

This file is for *behavior* (what to do / not do, where to look, what not to violate). For *how things work*, link into [DESIGN.md](DESIGN.md). For *what changed*, see [Tracker.md](Tracker.md).

---

## What this is

Lichborne is a DragonRealms (text MUD) game client. Electron + React + TypeScript. Connects via Lich5 (recommended) or direct SGE login. Windows x64 only. Single-developer project with a small, reachable tester pool.

**Product position (DESIGN.md §1, §24):** **display and configuration layer over Lich**, not a Lich-replacement. Lich owns automation (scripts, variables, triggers, repository). Lichborne owns rendering, themes, the panel system, profile portability, and surfacing Lich state. **Push back on requests that head toward duplicating Lich functionality** — say so before implementing.

---

## Principles (rules of the road)

1. **YAML is truth for user state. localStorage is a working copy.** New per-character settings need only a `scopedKey(character, ...)` write + `saveProfile()` call. Shared settings live in `_shared.yaml` and need typed entries in `SharedProfile` + `buildSharedProfile` + `importSharedProfile`.
2. **Lich is optional but recommended.** Every feature must work without Lich; Lich integration is additive.
3. **No silent data loss.** If a file can't be processed (future-version YAML, malformed XML), log it and **preserve it on disk**. Never overwrite content the code doesn't understand.
4. **Theme any new UI.** All colors come from CSS custom properties in `themes.ts`. Use `color-mix(in srgb, var(--x) N%, transparent)` for tints. Check the light themes (Ivory, Mist, Parchment) and Terminal.
5. **Stream IDs preserve case end-to-end.** A Lich script that pushes to `LichScripts` lands in a panel called `LichScripts`. The `NEVER_DISCOVER` filter is the only case-insensitive check.
6. **Per-session everything.** Every IPC payload carries `sessionId`; every listener filters by it; every per-character state lives under `lichborne.{character}.{suffix}`. Cross-session bleed is the most common bug class.
7. **Fix root causes, not symptoms.** Past root-cause fixes (B14, B17, B20, B33, B36, B53, etc.) left inline comments explaining the current shape of the code. Read them before patching nearby logic.
8. **Ship reversibly.** Bold changes welcome (v0.6.3 maps rewrite, login redesign). Pair them with migration paths (profile schema versioning, NSIS rescue hook) so a bad rev never costs a tester their data.

## Architecture (pointers, not duplication)

### Process split
- **`src/main/`** — Electron main (Node). Sockets, XML parsing, Lich child process, file I/O. Built with esbuild (`build-main.mjs`).
- **`src/renderer/`** — React UI. Built with Vite. Talks to main only via `window.api.*` (preload.ts contextBridge).
- **`src/shared/types.ts`** — single source of truth for IPC payloads and game-event shapes. Imported by both sides.

### Multi-session model (DESIGN.md §13)
One window, N character sessions. `src/main/main.ts` holds `sessions = Map<SessionId, Session>`; each Session owns its own `ConnectionManager`, `StormFrontParser`, `LichBridge`. Every push channel (`game-event`, `connection-status`, `raw-xml`, `error`) carries `sessionId`. **When adding any IPC, include `sessionId` in the payload.**

### Lich launch & connect (v0.7.0)
`LichConnection.launch()` spawns `ruby lich.rbw --stormfront --dragonrealms` as a **detached** child (not a service). Lich takes its force-mode path: bind `127.0.0.1:11024`, then `accept` **exactly one** front-end and **close the listener** — one Lich = one session. Readiness is **not** a fixed delay: `connectWithRetry()` retries the real connection (250ms cadence, ≥30s cap) until the port accepts — the first success IS the session socket (no throwaway probes). `launch()` resolves on the child's `spawn` event and watches `exit` so a Lich that dies on startup fails fast (with its piped stderr tail). Because each Lich serves one front-end then frees 11024, multi-character logins **must serialize** their spawn→connect window — `serializeLichLaunch()` in [ConnectionManager.ts](src/main/connection/ConnectionManager.ts) is a module-level promise chain shared across all per-session managers. SGE/eaccess auth runs outside that chain so it overlaps the queue wait. A failed launch `killProcess()`es its Lich so it can't squat the port.

**Known limitation — Ruby GTK script windows don't render.** Lich scripts that build their own OS-level UI via `Gtk.queue { ... }` + `Gtk::Window.new` (the `kill-counter.lic` shape — uses `Gtk::Box` / `Gtk::Entry` widgets and calls `window.show_all`) **silently fail to paint a window under Lichborne**. Root cause: force-mode (`--stormfront` flag) skips Lich's entry GUI, which is what starts the GTK main loop. The ruby-gtk3 gem still loads (so `HAVE_GTK` is `true` and the script's startup gate doesn't trip), but `Gtk.queue` blocks sit in a queue that nothing pumps — script's non-UI logic (counter increments, line filters, `_respond` calls) runs normally; UI work never executes. From the script's perspective everything is fine; from the user's perspective nothing happens. This is the same behaviour Stormfront the client exhibits when launching Lich in `--stormfront` mode (the path users typically work around by launching Lich standalone first), and it's not a Lichborne bug per se — it's a property of the launch model. **Product decision (2026-05-30, Sekmeht)**: do NOT add support. Three theoretical paths exist (detect-and-warn, change launch model to keep GTK main loop alive, document-and-steer) — all evaluated and declined. Lichborne is a display/configuration layer over Lich (Principle: not a Lich-replacement); GTK widgets are a Lich-runtime affordance that lives outside the front-end protocol Lichborne speaks. **When a tester reports "this Lich script doesn't work in Lichborne":** ask whether the script uses `Gtk.queue` / `Gtk::Window`. If yes, this is expected behaviour — point the script author at the `<streamWindow>` / `<pushStream>` XML approach (see `newkill-counter.lic` for the minimal pattern: `_respond("<streamWindow id='X' title='X' />")` + `_respond("<pushStream id='X'/>content<popStream/>")`), which Lichborne renders as a regular panel and which also works in every other front-end. Don't accept the symptom as a Lichborne bug.

### Game-text pipeline
`SGEConnection`/`LichConnection` → `ConnectionManager` (line events) → `LichBridge.interceptLine` → `StormFrontParser.parse` (stateful XML tokenizer) → `GameEvent[]` → batched IPC → `GameWindow`.

`StormFrontParser` is **stateful** (boldDepth, color stack, monoMode, injuryBuf, etc.). Call `parser.reset()` on new login. `<prompt>` is a good boundary for transient state.

Main text rendering uses **react-virtuoso**. Scroll pinning is a delicate state machine: `pinnedRef` snapshots before commit, `suppressUntilRef` gates programmatic scrolls (200ms; `scrollToBottom` 300ms), `onWheel` un-pins synchronously. Scroll-to-bottom is **instant** (`followOutput` returns `'auto'`). A story-window smooth-scroll feature existed v0.6.8–v0.6.11 (opt-in setting + flood-adaptive fallback) and was **removed in v0.6.12** — marginal, and a magnet for false bug reports. **Don't add `useEffect`-driven scroll logic** (or re-add smooth scroll) without reading GameWindow.tsx and DESIGN.md §23. Room-state updates funnel through the rAF pump (`roomQueueRef`), not direct `setRoomState` — see DESIGN.md §23 and pitfall #20.

### Profile system (DESIGN.md §20)
YAML files in `app.getPath('userData')/profiles/` — `%APPDATA%\lichborne\profiles\` in production (lowercase — `app.getName()` reads the package.json `name` field, NOT `build.productName`). `ensureProfilesDir()` in [profiles.ts](src/main/profiles.ts) is the canonical accessor. Two-stage migration (NSIS `preInit` + runtime fallback) rescues pre-v0.6.4 install-dir profiles.

- **Per-character setting** — write to `scopedKey(character, 'myThing')` and call `saveProfile()`. The dynamic `state:` pipeline picks it up automatically. No type changes needed.
- **Shared setting** — must be added to `SharedProfile` type + `buildSharedProfile` + `importSharedProfile`.
- **Schema versioning** — both files declare `profileVersion`. Migrations registered in `profile-migrations.ts` by SOURCE version. Adding optional fields is NOT a breaking change. Future-version YAMLs are skipped (preserved on disk).
- **Backups** — timestamped `{name}.yaml.{ISO-ish}.bak`, rolling retention of 5.

### Map system (DESIGN.md §19)
Two views: **Lich Map** (Lich image tiles) and **Genie Maps** (Genie XML-driven graph). Genie XML is the spatial source of truth for the graph view — coordinates come directly from the hand-curated XML files; no auto-layout, no BFS placement, no zone stitching. One zone visible at a time, picked by the dropdown or auto-switched when the player's room title matches a room in another loaded zone.

**Two prior approaches have been deleted; do not reintroduce.** The per-zone Genie Graph (`MapGraphView.tsx`, deleted v0.6.3) tried to render zones individually with the old augmentation matcher. The Lich-native BFS view (`LichGraphView.tsx` + `lichLayout.ts`, deleted v0.6.6) tried to make every Lich room renderable by inferring layout from `wayto` cardinals — but Genie's hand-curated coords and Lich's directional walks disagree in dense clusters, producing "type west, marker goes north" misrenders.

**Rendering matches Genie's `MapForm.cs` exactly.** 8×8 node rects are CENTERED on the XML position because Genie's `ConvertPoint(pos, 4)` SUBTRACTS the offset (MapForm.cs:187–193). Arc endpoints go through the XML position directly. Labels at `(pos + 1, pos + 1)` matching `DrawString(r.X + 1, r.Y + 1)`. Getting any of this wrong visibly misaligns label clusters; verify against the reference source instead of guessing.

**Arcs render in two passes** for dense-cluster legibility: under-pass at full opacity (drawn before nodes; lines disappear into rect fills inside clusters) + over-pass at 0.35 opacity drawn on top (faint trace across rects so you can follow a line all the way to its endpoint). Each pass collapses N arcs into 3 `<path>` elements (one per cardinal/climb/go category) via concatenated `M x,y L x,y` segments; same data, render twice, ~6 SVG elements total.

**Cross-zone stubs** are 1-room boundary markers with `note="MapXX_Name.xml"`. `isStubNode(n)` detects `.xml` in note aliases. Title lookup MUST prefer non-stub matches (otherwise the marker in zone A wins over the real room in zone B) — and it merges `byTitle ∪ byNormalized` *before* the non-stub filter, so a `byTitle`-only stub can't trap the match when the real room is reachable through the normalized form (B78).

**Ambiguous-title disambiguation in `currentLocation` (v0.7.0).** Same-titled rooms (the Crossing has 7 "Moonstone Street" rooms) resolve in this order: (1) exact normalized **description** match; (2) **graph adjacency** — the candidate joined by a Genie arc to `prevLocRef` (the last resolved room); (3) file order (`pool[0]`) as last resort. Adjacency is what makes the marker track correctly *while running* — the game streams room titles with no fresh description per step, so without it the matcher fell back to `pool[0]` and the marker stuck on the wrong same-named room until the player typed `look`. `prevLocRef` is updated one step behind by an effect, advanced only on a non-null match.

**Click model (v0.6.8): left-click pins, right-click walks.** Left-click a regular room → toggles a pinned gold BFS path (no walking). Left-click a stub → switches the displayed zone to its target XML. Right-click anything → walks (BFS in-zone, `sendWalkPath`). The map only auto-switches zones when the player's `roomTitle` actually lands in the new zone — never on walk completion (the game can block any command; racing the map ahead strands the user).

**Camera follow-the-player** uses `useLayoutEffect`, not `useEffect`, so the transform update lands in the same paint frame as the indicator's new world position. Default ON; manual pan/zoom turns it off; the ◆ button re-enables. Always-center model (not margin-snap). The follow effect gates on `followNode` — the *same* `visibleById.get(...)` lookup the indicator uses — so "marker visible" and "camera following" can't desync.

**Smooth camera motion.** Pan group is positioned with the CSS `transform` *property* (not the SVG attribute) so `.genie-pan-smooth` can `transition` it (`150ms linear` — linear, not ease-out, because a follow camera re-targets every step and ease-out's velocity reset pulses visibly). `snapTransform` (delta > 600px or > 20% scale vs last-painted transform) drops the transition for big jumps so zone switches / ◆ / fit cut instantly. The indicator carries the *same* transition + snap flag so it stays screen-centred during walks instead of bouncing. Gated on the `mapAnimations` setting (default ON): pan group + indicator only get `.genie-pan-smooth` when `!isDragging && mapAnimations`. (v0.6.8–v0.6.11 gated this on a separate `smoothScroll` setting; v0.6.12 removed that and folded the map glide under `mapAnimations`.)

**Current/selected/hover/hover-path/pinned-path indicators are hoisted OUT of `nodeRects`** as single overlay elements. Pre-hoist, walking re-built the entire `nodeRects` JSX every step. The current-room indicator is a sonar locator: two `genie-here-ping` expanding rings + the solid `INDICATOR_R` ring; the pings are exempt from the animation-pause via a higher-specificity rule.

**Per-color effect system (v0.6.7).** Every COLOR_LEGEND category has its own visual signature (heartbeat for Healer, coin glint for Shop, ripples for Water, leaves for Lumberjacking, dirt for Mining + Trailhead, vortex/drift/rise sparkles for the magical 4, etc.). The structure is uniform — adding a new effect is a 4-step pattern: (1) declare a `*_COLORS` set or extend `MOTE_EFFECTS`, (2) add a memo at the appropriate place in the component that builds the SVG elements per visible-node match, (3) add a CSS keyframe in `map-panel.css`, (4) render the memo at the correct layer in the SVG tree. Stub `↗` glyph always wins over category glyphs (cross-zone identity > resource category). `getMoteContrastColor(hex)` picks mote fills based on room luminance (light bg → dark motes, dark bg → light motes). `normalizeNodeColor()` in `parseGenieZone` converts CSS color names (`"Blue"`, `"Red"`, etc.) to canonical hex so downstream effect lookups don't silently miss rooms.

**Animation pause — two freeze classes (v0.6.7, extended v0.6.9).** The pan group `<g>` carries ONE of two mutually-exclusive freeze classes, both cascading `animation-play-state: paused` to descendants:
- `genie-pan-dragging` — *transient* freeze when `isDragging || inMotion`. `isDragging` flips on mousedown/up; `inMotion` flips true on any `currentLocation` change and resets a `MOTION_QUIET_MS = 800` timer. The locator sonar ping is **exempt** (`.genie-pan-dragging .genie-here-ping` keeps it running) — you still want to see yourself mid-walk.
- `genie-anim-off` — *permanent* freeze when `settings.mapAnimations` is off (default ON). **No ping exemption** — "off" stops the ping too.
The className picks one or the other (`!mapAnimations ? 'genie-anim-off' : (isDragging||inMotion) ? 'genie-pan-dragging' : ''`) so the ping-pause and ping-exemption rules never collide.

**Category effects are OMITTED (not paused) and CAPPED (v0.7.0).** Two layers of defense, both because SVG CSS animations are expensive — each animated element forces Recalculate Style every frame and is a compositor-layer candidate:
- *Omitted during motion.* `animation-play-state: paused` stops an animation advancing but leaves the element layer-promoted — profiling during travel showed residual Layerize ~21% even with effects "frozen". So the 10 animated per-category effect groups (sparkles, heartbeats, coin glints, ripples, bubbles, caution/implode rings, leaf/dirt falls, XP rises) render only when `showEffects = mapAnimations && !isDragging && !inMotion`. During travel/pan/off they're not in the DOM at all. They re-mount once the player has been still for `MOTION_QUIET_MS` (1500ms — long enough that normal sub-second running cadence never re-mounts mid-run; a short window would tear them up and down every step).
- *Capped to the nearest N rooms.* Idle in a dense zone (the Crossing) all effects on was ~29% Recalculate Style. The effect memos iterate `nearbyNodes`, not `visibleNodes` — `EFFECT_CAP` (30) rooms nearest the player (or zone-centre when viewing a zone you're not in), or the stable `EMPTY_NODES` ref when `!showEffects`. So at most ~30 rooms animate regardless of zone density, and the effect memos don't recompute during travel (dep is the unchanging `EMPTY_NODES`). **`auras` deliberately still uses `visibleNodes`** — they're static (no animation, no per-frame cost) and they ARE the colour map.

The freeze classes still cover what *stays* rendered (the sonar ping, the static/fire auras). Node **hover** is likewise suppressed during motion (`onNodeHoverEnter` checks `inMotionRef`) — the map scrolling under a stationary cursor otherwise fires a storm of pointer enter/leave → re-renders.

**`mapAnimations` setting (v0.6.9; sole map-motion switch as of v0.6.12).** `mapAnimations` (default ON) is the single toggle for ALL Genie map motion — the per-color category effects AND the camera glide. Off → `genie-anim-off` freezes effects and the glide is disabled (camera snaps). Per-character `AppSettings` field, persists via the dynamic profile `state:` pipeline, threads `GameWindow → PanelFrame → MapPanel → GenieMapView` (the `autoLinkUrls` path). (v0.6.9 also had a `smoothScroll` setting; removed v0.6.12 with the story-window smooth scroll.)

**Genie parse cache (v0.6.7).** First-time parse of a 122-XML maps folder takes several seconds (DOMParser is synchronous and chunky). `genie-cache:load` / `genie-cache:save` IPC handlers in `main.ts` serialize the parsed `Map<zoneId, GenieZone>` to `userData/genie-cache.json`. Subsequent launches verify a fingerprint (sorted `filename:mtimeMs:size`) against current folder state and skip re-parsing if it matches — load drops from seconds to ~50ms. Version field (`GENIE_CACHE_VERSION`) lets us invalidate every user's cache when the GenieZone shape changes without manual cleanup. Cache write is fire-and-forget after parse.

Files: [MapPanel.tsx](src/renderer/components/panels/MapPanel.tsx), [MapImageView.tsx](src/renderer/components/panels/MapImageView.tsx), [GenieMapView.tsx](src/renderer/components/panels/GenieMapView.tsx), [mapTypes.ts](src/renderer/components/panels/mapTypes.ts).

### Theme system (DESIGN.md §6)
~150 CSS custom properties in `themes.ts`. `darkBase` is the canonical full set; every other theme is a partial override merged over `darkBase` in `applyTheme` / `applyCustomTheme`. Theme Editor (`ThemeEditor.tsx`) uses a `TABS` declarative schema — new variables surface there.

### Automations, groups, modes (DESIGN.md §8)
Highlights, triggers, macros, aliases, and contact templates share a `groupIds: string[]` + `allGroups: boolean` shape. `isRuleActive(rule, activeGroups)` gates them. New rule types should follow this pattern and default to `allGroups: true`.

### Login & character-selection flow (v0.8.0, DESIGN.md §13.6)
Empty state and "+" tab open `Launcher.tsx` + `AddCharacterWizard.tsx`, not `LoginScreen.tsx` (kept only as type-host for `SessionInfo`). `LichSetupFields.tsx` is the single source of truth for Lich path/port/mode editing — three render sites (Launcher dialog, Settings panel, legacy LoginScreen), all must wrap it in `.login-form` class for CSS scoping.

- **Launcher rendering rules** are documented in DESIGN.md §13.6. Key invariants: Favorites is always-open; per-account sections default collapsed but **single-account is always expanded** and **freshly-added accounts auto-expand once** (wizard writes the account name to `lichborne.launcher.expandedAccounts` before `onCompleted` bumps `refreshKey`, and Launcher's refresh effect re-reads it).
- **AddCharacterWizard is account-driven, not character-driven** (v0.8.0). Step 1 collects account/password/game; step 2 runs `eaccessFetchCharacters` (mode-agnostic SimuCo auth, no Lich needed) and bulk-writes one stub `CharacterProfile` per checked discovered character. Existing profiles are listed but disabled. No connection happens in the wizard — tiles appear, user clicks Connect when ready.
- **Per-tile state is launcher-owned and writes immediately** via small read-modify-write helpers (`setCharacterFavorite`, `setCharacterGame`, `setCharacterUseLich`, `setCharacterHidden`, `patchCharacterProfile` for the notes editor). Never go through the GameWindow's debounced `scheduleProfileSave` for these — that's a different ownership track (see Pitfall #26 below).
- **Bulk Connect** ([BulkConnectPicker.tsx](src/renderer/components/BulkConnectPicker.tsx) + `runBulkConnect` in App.tsx) runs sequential per-account connects. DR's one-character-per-account rule forces this; sequential is also the simpler error-isolation model.
- **Same-account conflict** raises a confirmation modal (Cancel / Disconnect-and-continue) instead of refusing. The disconnect uses the awaitable `disconnect-await` IPC; the conflicting tab stays in the bar in disconnected state.
- **Shutdown** uses `gracefulDisconnect({ quickClose: true })` which calls `socket.end()` (guaranteed flush of QUIT) + force-close without waiting for server ack. ~300ms total regardless of session count. In-tab Disconnect and the conflict-modal auto-disconnect keep the full 5s ack-wait — they need the account slot confirmed released for the next action.

### Import wizard (DESIGN.md §15)
Wrayth XML / Genie .cfg / Frostbite .ini parse into a neutral `ImportCandidate` intermediate, then `mapper.ts` converts to native rules with `allGroups: true` and blank names. Per-client quirks under `src/renderer/import/`. Substitutions counted but not imported (F12).

### Session Log (DESIGN.md §28, shipped v0.7.0)
Per-character daily plain-text logs. **Capture** happens in `GameWindow.onGameEvent` → `logToSession()` → `session-log:append` IPC → buffered per-character writer in [sessionLog.ts](src/main/sessionLog.ts). Files: `{userData}/Logs/{Character}/{Character}_YYYY-MM-DD.log`, line format `[HH:MM:SS][stream] text` (the date is in the filename; parsers also accept the legacy `[YYYY-MM-DD HH:MM:SS.mmm]` form). Buffered (1s / 100-record flush, force-flush at 5000); `flushAllSessionLogs()` runs on graceful close next to `backupAllProfiles()`.

- **Config is app-wide, NOT per-character.** All Session Log preferences (capture gates, retention, compression, raw-size cap, the Recent-tail filter, the Export-builder format prefs) live in one `SessionLogSettings` object — [sessionLogSettings.ts](src/renderer/sessionLogSettings.ts), stored in `_shared.yaml` via `SharedProfile.sessionLog`. `logToSession` calls `loadSessionLogSettings()` fresh per batch (it's a tiny localStorage read — no stale-closure ref needed, and a Settings change applies instantly to every open character). The capture config also rides along in `SessionLogAppendPayload` (`retentionDays`/`compress`/`maxRawMB`) so main can run maintenance without its own copy. Both the SettingsPanel section and the modal read-modify-write the shared object so neither clobbers the other's fields.
- **What's skipped:** `raw`, room sub-streams (`ROOM_STREAMS`), blank lines — state, not history. Stream IDs `cmd`/`sys`/`main` are the capture-category keys; everything else counts as "streams".
- **Size management (v0.7.0):** closed (non-today) day-files are gzip-compressed to `.log.gz` (~85-90% smaller) by background `maintain()` — runs at session start + day-rollover, gated by `sessionLogCompress`. All readers route through `readLogFile()` (gunzips transparently) and `resolveDayFile()`. Two retention limits: `pruneOldLogs` (age, `.log` + `.log.gz`) and `enforceRawSizeCap` (caps *uncompressed* `.log` bytes only — never `.gz`, never today's file). If you change the line format or add a maintenance limit, keep both parsers (`LINE_RE` in the modal, `EXPORT_LINE_RE` in main) accepting the legacy form.
- **UI:** [SessionLogModal.tsx](src/renderer/components/SessionLogModal.tsx) — one modal, three views (Recent Tail + Quick Search + Export), opened by the toolbar **Logs** button or right-click → "Show in Log" (which opens straight into Search). Tail is paginated (`PAGE`/`JUMP_RADIUS`); never loads a whole day. The Recent-tail filter (`filterHidden`/`filterDedup`) and the Export format prefs persist in the app-wide `SessionLogSettings`, not per-character.
- **Export builder ("Create Log File"):** the renderer gathers a `SessionLogExportSpec` (date range, stream layers, include-timestamps/tags, dedup, summary, split-per-stream, target) and main does all the reading/filtering/formatting/writing in `session-log:build-export`. Big data never crosses IPC — only the spec and a small `SessionLogExportResult`. `target: 'file'` runs a save (or folder, when `splitPerStream`) dialog; `'clipboard'` writes combined text via Electron `clipboard`.
- **Read IPC:** `session-log:list-days`, `read-day` (tail + `beforeLine` pagination), `list-streams` (single day, or a range when `toDate` is passed), `search` (returns `SessionLogSearchHit` with `lineNo`/`total` for jump-to-tail), `build-export`, `disk-usage`, `open-folder`. All read handlers flush the character's buffer first so reads reflect the live session.

## Pitfalls (read before touching adjacent code)

> **Adding a new pitfall — read this first.** Pitfalls are numbered sequentially and the numbers ARE referenced from other comments / commit messages / docs (e.g. "see CLAUDE.md pitfall #33"). When adding a new one, **find the highest-numbered existing pitfall and add the new one AFTER it, with the next number** — do NOT insert before an existing pitfall or renumber an existing one. The natural editing instinct (using Edit with an existing pitfall as the anchor `old_string` and putting the new content first in `new_string`) puts the new pitfall in the wrong position; the correct anchor is the `---` separator at the END of the pitfalls list, with the new pitfall inserted between the last existing pitfall and that separator. Sekmeht caught me making this exact mistake three times across v0.8.7 and v0.8.8 (B121, B124, B126) — each time renumbering an older pitfall instead of just appending. The fix path when caught: revert the renumbered older pitfall to its original number, then append the new one at the end. The numbering is `1, 2, 3, ..., 31, 33, ...` (32 has been missing for ages; don't try to fix the gap, it's referenced from elsewhere by its current number).

1. **`app.getName()` returns `name`, not `productName`.** userData is `%APPDATA%\lichborne\` (lowercase). The `productName: "Lichborne"` under `build:` only affects installer / Start menu display. Mixing the cases caused a v0.6.4 hotfix to silently no-op.
2. **electron-builder NSIS `customInit` runs AFTER the previous uninstaller.** Use `preInit` and read the previous install location from `HKCU\Software\${UNINSTALL_APP_KEY}` (`$INSTDIR` isn't set that early).
3. **NEVER put user data inside the install directory.** NSIS `RMDir /r $INSTDIR` on every upgrade. Pre-v0.6.4 lost everyone's profiles this way.
4. **`StormFrontParser` is stateful and lives for the session lifetime.** Reset on new login. Stream-mode state can bleed across `<prompt>` boundaries.
5. **Scroll pinning fights you if you add naive useEffect logic.** Read GameWindow.tsx + DESIGN.md §23 before adding any programmatic scroll.
6. **Stream IDs are case-sensitive everywhere except `NEVER_DISCOVER`.** Lowercasing them anywhere will break Lich scripts. See B34.
7. **`lichPort` is the Lich front-end port, not an SGE shard port.** Don't look it up from a per-game table at credential build-time (v0.6.3 bug).
8. **`setState` in a mousemove handler must be gated on a hover sentinel** or you re-render on every pixel. See `GenieMapView.tsx` `onMouseMove` (gates on `dragRef.current`).
9. **Genie node rects are CENTERED on the XML position, not top-left anchored.** `ConvertPoint(pos, 4 * scale)` SUBTRACTS the offset (MapForm.cs:187–193). Anchoring top-left shifts every node 4px down-right and misaligns label clusters. Verify against Genie source for any map-rendering math change.

27. **`discoveredStreams` must never contain a builtin `PanelType` id.** A "discovered" stream is one the game emits via `stream-declare` / `stream-push` events; the renderer adds it to the `discoveredStreams` array so the Panel Manager can surface it as a Custom row in Available Streams. 28. **Never call `navigator.clipboard.writeText` from the renderer. 29. **Highlight / trigger regexes are compiled with `/g`; loops using `regex.exec()` must `lastIndex++` after a zero-width match.** Highlights compile via `new RegExp(...source, rule.caseSensitive ? 31. **`dispatchUserText` is the canonical "process text as if the user typed it" path; don't bypass it with raw `window.api.sendCommand` when you want alias resolution + echo + history + log.** Extracted from `handleCommand` in [GameWindow.tsx](src/renderer/components/GameWindow.tsx) as part of v0.8.3 F07 (repeat-command macros). The two callers are (a) the command-bar Enter handler — `{ pushToHistory: true, clearInput: true }`, and (b) the macro token expansion site (`{RepeatLast}` / `{RepeatSecondToLast}` / `{ReturnOrRepeatLast}`) — `pushToHistory` and `clearInput` vary by token. 33. **Accessibility overlays (high contrast, color blind) live on top of theme vars — never write theme vars without re-applying overlays.** `applySettingsToDOM` in [settings.ts](src/renderer/settings.ts) writes the active character's accessibility overlay vars *after* the theme vars (e.g. deuteranopia rewrites `--vital-health-ok-start` to a teal). 34. **Theme vars: panel-specific colors should cascade from the closest general `--*` var via `var(...)` in `darkBase`, not literal hex.** Established by B108 (`--map-bg` → `var(--bg-app)`) and broadened in B112 (8 panel text vars → `var(--text-secondary/muted/dim)`). The Theme Editor only exposes the *general* surface vars (App background, Game text, Labels, Muted text, etc.); custom themes built via the editor never set the specialty vars. With literal hex defaults in `darkBase`, custom themes were stuck with whatever value `darkBase` baked in — Rakkor's "Game Text" black on a light theme left Room / Hands / Exp panels stranded grey. Pattern when adding any new panel-specific text/bg/border var: pick the closest general var that matches the role (body text → `--text-primary` or `--text-muted`; section labels → `--text-secondary` or `--text-dim`; panel bg → `--bg-base`; panel surface bg → `--bg-app`) and default to `var(--general)`. Built-in themes (Parchment, Mist, Ruby, …) that need a distinct color *still* set the specialty var explicitly — `applyTheme` merges over `darkBase`, so an override always wins. Held / active accent colors (`--hand-held-color`, `--spell-active-color`, etc.) are the exception — they ARE the visual cue and should stay specialty-tinted. If you find yourself writing `'--new-thing-color': '#888888'` in `darkBase` and `#888` is also the value of `--text-muted`, that's the smell: write `'var(--text-muted)'` instead. If you find yourself adding a new specialty var to the Theme Editor schema *just so* light-theme users can override it, that's also the smell — make the cascade do the work.

36. **Contact tracking has TWO long-running effects (last-seen + time-spent polling) that BOTH share the same `pendingContactsRef` buffer — every read AND every state-change must respect it.** Established in B119 + F34. The last-seen effect (fires on `roomState.players` change) builds a 2-second debounced buffer in `pendingContactsRef` to batch lastSeen / lastRoom / encounterCount updates. The 60s `timeSpentMs` polling tick reads contacts to add 60s to anyone in the room. Both effects MUST start from `pendingContactsRef.current ?? contactsRef.current` — using `contactsRef.current` alone misses in-flight buffer updates and the next `saveContacts` clobbers them. The buffer also gets cleared whenever `contacts` state changes externally (ContactsPanel save, import, polling-tick flush): a no-deps `useEffect([contacts])` nulls `pendingContactsRef` + `clearTimeout(lastSeenTimerRef)` — without this clearing, an in-flight buffer would write a stale snapshot back to localStorage and wipe whatever the user just added (B119: Rakkor's "save 12th contact, it disappears"). When adding a new contact-tracking effect, always: (a) base on `pendingContactsRef.current ?? contactsRef.current`, (b) write through `saveContacts` + `setContacts` + `scheduleProfileSave` together, (c) trust the B119 cleanup useEffect to drop your buffer when state changes externally. Don't write to `contactsRef.current` directly — it's a downstream mirror, not a source of truth.

37. **Encounter / presence detection needs TWO gates, not one — absence-threshold AND cooldown.** F34's `encounterCount` increments only when a contact is detected AND `(now - lastSeen) > ABSENCE_THRESHOLD_MS` AND `(now - lastEncounterAt) > ENCOUNTER_COOLDOWN_MS`. The absence-threshold (90s, > polling-tick 60s with a margin) gates "they actually LEFT since last seen" — without it, a contact who stays in your room for 10+ minutes ticks the counter every time `room.players` changes (someone else enters / leaves) because the last-encounter timestamp is old. The cooldown (10 min) gates "even if they left, don't double-count a quick exit-then-re-enter." `lastEncounterAt` updates only on actual increments; `lastSeen` updates on every detection. Both timestamps are needed and serve different purposes — don't fold them. When adding any new contact-state-transition feature (e.g. "first seen on day X" badges), think in terms of presence-edge detection rather than just "is this contact here right now," and use the same absence-threshold + cooldown shape so we don't end up with multiple half-correct definitions of "encountered."

39. **Panel Manager has two ownership dimensions; don't conflate "slot added" with "slot has streams".** v0.8.1 introduced explicit add/remove of panel locations — the 4 slots (Main-Top, Top-Right, Middle-Right, Bottom-Right) each carry an `*Added` boolean (`mainTopAdded` / `topAdded` / `midAdded` / `bottomAdded`) that controls whether the slot renders in the game window. The slot's `tabs` array controls which streams live in it. The two are independent. (b) `allTabs` in [PanelManager.tsx](src/renderer/components/PanelManager.tsx) — used by `openCustomIds` and `openTypes`; without the gate, phantom tabs hide discovered streams from Available Streams, leaving the user with no path to re-add a stream to a visible slot (this was Legiro's moonwatch / Moons bug). Migration defaults when the flag is missing: **Main-Top → false** (new zone, explicit opt-in), other three → **true** (preserve v0.8.0 always-visible). When adding a fifth panel location in the future, follow the same pattern: new `*Added` state + scopedKey persistence + migration default + render-gate in GameWindow + `addPanelZone`/`removePanelZone` handlers + PanelManager UI entry + **audit every cross-zone aggregation site to gate on the new flag**.

40. **`newDiscovered` in GameWindow's event batch must dedupe WITHIN the batch, not just against existing state.** Established in B123 (v0.8.7). The event loop in [GameWindow.tsx](src/renderer/components/GameWindow.tsx) translates every incoming `stream-declare` AND `stream-push` event into a push to a per-batch `newDiscovered: string[]`. The dedup filter at the `setDiscoveredStreams` site checks `existing.has(id)` (against pre-batch state) but originally did NOT track ids it had already added in the same pass. A Lich script that emits `<streamWindow id='X'/>` immediately followed by `<pushStream id='X'/>` produces both events in a single batch; both push `'X'` into `newDiscovered`; both pass the `existing` check (since `X` isn't in `discoveredStreams` yet); both get appended → array becomes `['X', 'X']`. PanelManager renders one row per array entry, so the user sees two identical "Available Streams" rows; adding either makes `openCustomIds.has('X')` true and filters out BOTH rows (this is also how to recognize the bug — adding "one" of two identical-name rows hiding both is a duplicate-id signature, not a display bug). The fix is a `seenInBatch: Set<string>` local to the filter; first entry in a batch passes and gets recorded, duplicates within the same batch are rejected. When adding a new stream-discovery event type in the future (a hypothetical third tag that should also surface a stream), route it through the same `newDiscovered` push site so the existing dedup catches it for free. If you add a separate write site, replicate the `seenInBatch` pattern. The corresponding parser invariant: `stream-declare` (from `<streamWindow>`) and `stream-push` (from `<pushStream>`) deliberately emit independently — script writers' intent is "make sure the UI knows this stream exists" with multiple safety nets, and stripping one of them at the parser layer would break legitimate scripts. Dedup is the renderer's job.

41. **Room sub-stream stale-clear lives at the parser, not the GameWindow event batch.** Established in B121 (v0.8.7). The Room panel's structured sections (`room`, `room-objects`, `room-players`, `room-creatures`, `room-extra`, `room-exits`) get populated from `<component id='room ...'/>` tags. When DR doesn't re-emit a component for a room transition (common — DR is sparse with re-emits unless `LOOK` forces them), the previous room's data carries over and surfaces as ghost players / objects in the Room panel until a manual `LOOK`. **The wrong place to fix this is GameWindow's event batch loop** (which is where I tried first): track which sub-streams "were seen" in this batch, clear the rest on title change. The fatal failure: IPC batches can split title-change from component updates across multiple batches. A batch carrying only the new title with no component re-emits would falsely "see no players this batch" and wrongly clear data the next batch was about to populate. **The right place is the parser** ([StormFrontParser.ts](src/main/parser/StormFrontParser.ts)). DR's XML protocol orders `<streamWindow>` (which carries the subtitle = room title) BEFORE any `<component id='room ...'/>` for the new room — deterministic, single-batch-or-not. New `lastRoomTitle: string` field on the parser; in the streamWindow handler, when `cleanTitle !== lastRoomTitle` emit `clear-stream` for all six room sub-streams BEFORE the typed `room-title` event. Components that DO arrive then re-populate via their own clear+stream-text flow; sections that DON'T arrive stay correctly empty. The `lastRoomTitle` gate prevents a no-op streamWindow re-emit (same subtitle, DR sends these for non-transition repaints) from clobbering freshly-populated sub-stream data. **B117 compatibility**: the parser's `captureSegments` work for monsterbold lives inside the `<component>` handler which fires AFTER the streamWindow pre-emit clear-streams — bold capture is unaffected. **Downstream beneficiary — contacts tracking**: the F34 encounter / `timeSpentMs` effects iterate `roomState.players`, so before B121 they were ingesting carryover ghost players from the previous room and crediting time-spent against the wrong room. B121's clear at the parser cascades to a clean `roomState.players: []` during the transition gap, so contacts stats are now accurate by construction (no separate fix needed for F34). **Don't move this logic back into GameWindow** — every fix at that layer has to know about batch boundaries, which makes it brittle. The parser sees the raw tag stream and knows the ordering. **Reset on session boundary**: `lastRoomTitle` is cleared in `parser.reset()` alongside other per-session state, so the first room of a new session always triggers clear-streams.

42. **Lichborne-source imports bypass the `ImportCandidate` intermediate; new rule types must update the bypass, not just the intermediate.** Established in B124 (v0.8.7). The import wizard's neutral intermediate at [types.ts](src/renderer/import/types.ts) (`ImportHighlight` / `ImportTrigger` / `ImportMacro` / `ImportAlias`) was designed to model the union of Wrayth / Genie / Frostbite source formats — it does NOT model the full native Lichborne rule shape. `ImportHighlight` has no `bold` / `glow` / `glowColor` / `groupIds` / `allGroups` / `name` / `enabled`; `ImportTrigger` has no `gates` / `oneShot` / `cooldownSeconds` / `watchStream` / nested action ordering / `groupIds` / `allGroups` / `name` / `enabled`; `ImportMacro` / `ImportAlias` are similarly trimmed. When F29 (Lichborne→Lichborne) shipped in v0.8.4, it routed everything through this intermediate, and the mapper at [mapper.ts](src/renderer/import/mapper.ts) hardcoded defaults for every field the intermediate didn't carry — silent data loss on round-trip across every rule type. **The fix shape**: `ImportResult` carries an optional `nativeRules?: { highlights, triggers, macros, aliases }` block (loose `unknown[]` typing to avoid a circular import with the rule modules). The lichborne.ts parser populates it in lockstep with the ImportCandidate arrays, spreading the source rule object and overriding `id: nanoid()` (and for triggers, every nested `TriggerAction.id` regenerates too). [ImportWizard.tsx](src/renderer/components/ImportWizard.tsx)'s `doImport` checks `result.nativeRules` and prefers it over `mapImportResult(...)` output when present; non-Lichborne sources leave it undefined and fall through to the legacy mapper path. **When adding a new rule type** (e.g. a future "scene", "snippet", "schedule"…), the new type needs: (a) full shape in the native rule module with `id` + `name` + `enabled` + `groupIds` + `allGroups` like every existing one; (b) an entry in `LichborneExportFile` and the parser's `nativeRules` block at [lichborne.ts](src/renderer/import/parsers/lichborne.ts) that spreads the source and regenerates ids; (c) an apply branch in `doImport` mirroring the existing highlights/triggers/macros/aliases shape — content-based dedup key, append-merge filter, save call. The ImportCandidate intermediate is **only** for surfacing the rule in the wizard preview UI; if you don't need preview support for the new type, you don't need an ImportCandidate at all (just `nativeRules`). The cardinal sin is the reverse — adding the field to ImportCandidate + mapper but forgetting to extend `LichborneExportFile`'s parser shape, which would re-introduce the B124 loss pattern. **Don't trust loaded rules to have default fields** — the runtime stores assume rules saved by Lichborne have all required fields, but a hand-edited YAML or a malformed export could be missing them. Defensive normalization belongs at the `loadHighlights`/`loadTriggers`/etc. layer, not the import boundary; we don't have it today because exports always come from `loadHighlights(character)` which returns runtime-complete shapes. Add it only when a tester actually hits a malformed-import crash.

43. **When the user can manually scroll past where the scroll API thinks the bottom is, the bug is in LAYOUT, not scroll math.** Established in B122 across v0.8.7 → v0.8.8. The half-row clip at font 13+ ate three failed hypotheses before landing on a real fix: (a) v0.8.7 lowered a `totalListHeightChanged` threshold from 2px → 0.5px — didn't help; (b) v0.8.8 first attempt replaced raw `el.scrollTop = el.scrollHeight - el.clientHeight` with `virtuosoRef.current?.scrollToIndex({ align: 'end' })` on the theory Virtuoso's internal accounting differed — didn't help; (c) Sekmeht's diagnostic — "I can manually scroll down ~half a row past where the snap lands" — proved BOTH APIs agreed on the wrong target. When two independent scroll APIs converge on the same position but the DOM has scrollable content past that position, the issue is in CSS / layout, not in the math. The fix that worked: a `components.Footer` on Virtuoso (`<div style={{ height: 14 }} />`, fixed 14px, gated to `settings.fontSize >= 13`) — defensive layout slack that doesn't require knowing why the scroll APIs land short. **General principle**: scroll-snap math problems are usually solvable by adding layout slack rather than by chasing what the math is doing. If you're tempted to write a third hypothesis-based fix to scroll math, stop — add invisible padding below the content instead. Specific to this case: the 14px footer is **gated on font ≥ 13** because at font 12 and below row heights round to clean integer pixels and no clip occurs; the slack would just be empty space below the last row (visible gap above the vitals strip). **Pair fix shipped in same release**: a separate `useEffect([settings.fontSize, settings.lineHeight, settings.largePrint])` re-snaps via `scrollToIndex({ align: 'end' })` ONLY when `pinnedRef.current` was already true — font / line-height / large-print changes reshape every row's height, growing scrollHeight, leaving the previously-pinned scrollTop short of the new bottom. Bail on `!pinnedRef.current` so scrolled-up readers aren't yanked back. rAF before the snap so Virtuoso has a frame to re-measure rows at the new dimensions. The threshold check in `totalListHeightChanged` is left at 0.5 (responsive) but doesn't carry the load — the footer does.

44. **The Room panel must mirror the main scroll's styling for the same content.** Established in B117 (v0.8.5, monsterbold creatures), reinforced in v0.8.8 (line-mode highlights for treasures). DR's `<component id='room players'/>` / `room objs` / `room creatures` / `room extra` sections are the exact same content that appears in the main scroll's "You also see ...", "Also here: ...", and (for desc) the room paragraph. Anything that styles that content in the main scroll — server-pushed bold/color, user contact name highlights, user match-scope highlights, user line-mode highlights — should style the structured Room panel sections the same way. Two things flow from this: (1) **the parser's `captureSegments` must carry the same styling fields as `pushSegment`** (currently: `bold` from B117; future styling sources will need to be added to both). When you add a new styling field to TextSegment (a `class`, `linkType`, whatever), update both the main-scroll `pushSegment` AND the capture path in `text()`'s `captureSegments.push`. (2) **RoomPanel's `renderSegments` must pass every kind of rule that TextLineRow passes** — `matchRules` (B111), `lineRules` (Rakkor v0.8.8), contacts, contact templates, nameRegex. The line-style is computed PER-SECTION (objects/players/creatures/extra each compute their own `getLineHighlightStyle` off their own joined text) so a rule that matches a player name paints only the Players section, not the others. `desc` deliberately stays out of line-style scope because it's multi-sentence prose where a single match would over-paint a large block. **B111's original "lineRules excluded on purpose" exclusion was overcautious and got overturned in v0.8.8** — the documented "lobster paints the whole section" objection turned out to be the wrong call: in the main scroll the same content gets that same paint when the user authored a line-mode rule, and consistency between surfaces beat the theoretical objection. Future "should the Room panel apply X?" questions should default to YES if the main scroll applies X for the same content.

45. **UI that should scale with `Settings → Font Size` must use `em` (inheriting from `.game-toolbar`) or `var(--game-font-size)` directly. Never `rem` for game-area UI.** Established in B113 (text weight pass), reinforced in v0.8.8 Mode button fix. `rem` is anchored to the browser root font-size (~16px) and ignores the user's per-character `settings.fontSize`. So a button declared as `font-size: 0.75rem` renders at a fixed 12px regardless of whether the user picked 8 or 24 in Settings. Three sizing categories in the codebase: **(a) toolbar buttons** like `.btn-debug` / `.btn-theme` / `.btn-settings` use `font-size: 1.0em` and inherit from `.game-toolbar` (which itself scales). New toolbar buttons should match this pattern — the Mode button shipped with `0.75rem` and visibly drifted smaller as users bumped their font size; v0.8.8 fixed it to `1.0em`. **(b) Portaled popovers** (Mode switcher menu, contact popover) can't inherit from `.game-toolbar` because they're mounted to `document.body` via React portal. Use `var(--game-font-size)` directly with a multiplier (e.g. `calc(var(--game-font-size) * 0.85)` for smaller sub-text). **(c) Text-line content** uses `var(--panel-font-size, var(--game-font-size))` to support the F31 per-panel override with a fallback to the global. When adding any new UI surface in the game area, grep for `rem` in the new CSS before committing — every `rem` is a fixed pixel size that won't track user preference. App chrome (settings panels, modal headers, etc.) can use `rem` deliberately if it should NOT scale with game font, but that's the exception, not the default.

46. **DR's room-transition data arrives in TWO independent XML tags; the parser must read both, not just one.** Established in B126 (v0.8.8). When the player moves to a new room, DR usually emits both: (a) `<streamWindow id='main' subtitle='[Title - rmId]'/>` — carries the room title and room id together inside the subtitle bracket, parsed in [StormFrontParser.ts:441-484](src/main/parser/StormFrontParser.ts#L441-L484), and (b) `<nav rm='X'/>` — carries the new room id as an attribute, parsed in [StormFrontParser.ts:523-548](src/main/parser/StormFrontParser.ts#L523-L548). For most transitions both arrive together so reading just `<streamWindow>` is sufficient. **But some transition shapes (teleports, NPC-induced moves, certain scripted moves) emit ONLY `<nav rm>` and skip `<streamWindow>` entirely.** For those, a front-end that reads only `<streamWindow>` will have `roomState.title` and `roomState.roomId` frozen on the prior room until a manual `LOOK` forces a re-emit. Lich's `$room` variable is correct the whole time because Lich's scripting reads `attrs.rm` from `<nav>` independently — that's the canary: when `;e $room` shows the right room but the Lichborne map shows the wrong one, we missed a `<nav rm>` somewhere. **The fix shape**: a separate `RoomIdEvent` (not an overloaded `RoomTitleEvent`) — emit `{ type: 'room-id', roomId: N }` from the `<nav>` handler when `attrs.rm` is numeric. The handler in GameWindow routes it into `roomUpdates.roomId` ONLY — leaves title / desc / sub-streams alone, because forging an empty string for fields we don't have would wipe the only data we have (the prior room's title is at least *something* to display until a real `<streamWindow>` arrives). **Don't overload RoomTitleEvent with an empty title for the `<nav>` case** — `GameWindow`'s `case 'room-title'` does `roomUpdates.title = evt.title` unconditionally; an empty title would clear the prior title and the Room panel would go blank. Two events with different invariants is the right separation. **Scope of the Lich Map fix**: it's not a fix for the Room panel's title-display (only `<streamWindow>` carries the title string), and not a fix for Genie Map (its `currentLocation` memo is title-based, not roomId-based; B110's `staleZoneCountRef` escape valve still handles Genie's variant of the symptom). **General rule**: when extracting data from a stateful XML stream, audit every place the same data could appear. Room id is in two places (streamWindow subtitle + nav rm); both need handlers. The same audit should apply for any future "data appears in multiple tags" case.

## Proactive bug-checking

The goal is to find bugs before testers do — both new bugs the current change might introduce, and **latent bugs from prior versions that are still living in the current code**. Lichborne is single-developer with a small tester pool; every bug we ship is one a tester has to find and report, costing them and us time. Two complementary practices.

### Iron rule: read the actual code, not memory or docs

Every claim you make about how the code behaves must come from reading the file you're claiming about, IN THIS SESSION. Not from CLAUDE.md, not from pitfalls, not from a Tracker.md entry describing the code, not from "I remember how this works." Documentation describes the code's intent and history; only the source describes what it currently does. The B114 false-positive that prompted this rule (v0.8.7 retroactive sweep): pitfall #33 described the theme-overlay re-apply hook pattern, I worked from that description, claimed a multi-character bug existed, and was wrong — the actual effect at [GameWindow.tsx:954-973](src/renderer/components/GameWindow.tsx#L954-L973) had `isActive` in its deps and re-applied overlays directly on tab switch, which the pitfall didn't mention because the pitfall was scoped to a different scenario. Acting on the false positive would have shipped a redundant fix and possibly introduced a real new bug. The fix-cost of a false positive is higher than the fix-cost of doing the read.

**Verification protocol** when claiming a bug exists or doesn't exist:
- Grep for the specific function / state / effect you're reasoning about.
- Read the relevant lines yourself (Read tool with line offsets, not just the grep snippet — grep context can mislead by hiding deps arrays, guard clauses, or cleanup returns).
- Confirm the deps array of any `useEffect` you reason about — pitfalls describe patterns, not necessarily the current deps list.
- Confirm the cleanup function — pitfalls describe what's supposed to happen, not necessarily what does.
- If a pitfall and the current code disagree, **the code wins** — and the pitfall needs updating.

This rule has no "n/a." Every retroactive sweep claim, every "this is safe because…" assertion, every "no concerns" sign-off must be backed by a read in this session. The cost is 30 seconds of grep + read per claim; the benefit is not shipping false-positive fixes (and not missing real bugs that contradict your memory of how something works).

### Pre-merge checks

Before declaring a change done, run through these three questions. ~30 seconds per change, catches the classes of bug this codebase has actually shipped (B114 cross-tab overlay drift, B121 cross-feature state writes, the kind of stale-state-vs-new-code mismatches that come from incremental migrations). The right answer to any of these can be "n/a" — the point is to have considered it.

1. **What state does this write that other code reads?** List every var / ref / state your change writes. For each, grep one consumer and read it. Cross-feature state is where the bug class lives: B121 cleared `roomState.desc` and three other subsystems (Room panel, Lich Map findRoom, Genie Map memo, contacts effect) read it. The pre-check found the same-batch behavior was already absorbed by existing protections (B109 timer, F34 early-return); the after-the-fact discovery would have shipped as B12X. If your change writes nothing other code reads, the answer is "n/a" — say so.

2. **What cleanup runs on unmount vs. on hide?** Tabs in Lichborne don't unmount when you switch — they go `display:none` (pitfall #24). So `useEffect` cleanup only fires on real unmount (close session), not on hide. Anything that should re-fire on *show* (re-apply per-character settings, refresh layout-dependent measurements, reset transient state) needs an explicit activeId-watcher in App.tsx, NOT just a mount-effect in GameWindow. B114 shipped with this gap: theme-overlay re-apply only ran on theme change, not on tab switch, so two characters with different accessibility overlays bleed into each other. If your change relies on a mount/unmount lifecycle, ask "does this need to fire on show/hide too?"

3. **Does the current code handle state written by older versions of itself?** YAML profile files, localStorage entries, the genie parse cache, the panel layout state, custom themes — all of these were written by some prior Lichborne version a user already ran. When your change reads any of this data, what happens if the field is missing (older write), has a different shape (intermediate refactor), or has a value the new code doesn't expect? **Backward compatibility we care about runs in one direction only**: current code reading prior-version state. We do NOT care about older clients reading newer files — testers run the current build, not v0.8.4. The right answers: (a) treat missing fields as their type default (most optional YAML), (b) bump GENIE_CACHE_VERSION when the cached shape changes (pitfall #16), (c) write an explicit migration in `profile-migrations.ts` by SOURCE version when a field is renamed / restructured. The wrong answer is reading stale state and crashing / silently dropping data the user previously saved.

These are not a substitute for testing — they're a substitute for the bug report you'd otherwise get next week. The right output is either "no concerns" + a one-line note in the commit message, or a tracked issue if a concern surfaces that's out of scope for the current change.

### Retroactive sweeps at version boundaries

When cutting a new version (or whenever there's a natural pause), do a focused read of the last 3–5 versions' Tracker entries with a different question: **not "did the change work" but "what could a tester still hit that hasn't been reported yet?"** The pattern that ships dormant bugs in this codebase:

- A new feature (Fxx) writes state or uses an effect lifecycle in a way that's correct for its own scope but interacts unexpectedly with an OLDER subsystem the author wasn't focused on.
- The bug is silent — no exception, no obvious wrong rendering. Just a corner case where state drifts (B114), focus jumps unexpectedly (F32 edge cases), or stats accumulate against the wrong target.
- It takes a specific user configuration to trigger: multi-character + accessibility overlay (B114), Ctrl+Tab from a modal text input (F32), Lich script with both `<streamWindow>` and `<pushStream>` for the same id (B123).

For each recent feature, ask the three pre-merge questions retroactively. Read the consumers of any state it writes. Verify the cleanup story on show/hide. Check whether older saved state still loads correctly. Don't refactor on speculation — log a real concern with a repro path, or call it clean. The retroactive sweep done for v0.8.7 found B114 (real, fixed in v0.8.7) and noted F32 edge cases (deferred, not fix-worthy without a tester report). That's the right output shape: distinguish "real bug worth fixing" from "future-feature concern" from "minor papercut without a clean fix" and don't conflate them.

**When to do this**: at the cut of every minor version (v0.x.0 → v0.x+1.0) at minimum, and ideally before each patch release when a meaningful feature shipped recently. The cost is one focused review session; the alternative is a tester finding it cold and the trust hit that comes with shipping obvious-in-retrospect bugs.

## Operational guardrails

- **Never `git commit` without explicit user approval.** Asking is mandatory.
- **Windows + PowerShell** is the host shell. Use PowerShell syntax (`$null`, `$env:VAR`). Bash available for POSIX scripts.
- `profiles/` is gitignored; never commit user data captured during testing.
- `passwords.json` lives in `app.getPath('userData')`, encrypted with Electron `safeStorage` (Windows DPAPI). Don't move it into the repo tree.
- Prefer fixing root causes over adding fallbacks.
- **When proposing a feature, check the principles first.** If it duplicates Lich functionality, say so and ask. If it lives in the display/configuration/profile space, propose freely.
