# CLAUDE.md

AI-facing project guide. Loaded into every Claude session for this repo — keep tight, keep operational, keep current. **If you make a non-trivial architectural change, update this file.** The goal is that a fresh session can pick up the project without re-deriving its conventions from the code.

This file is for *behavior* (what to do / not do, where to look, what not to violate). For *how things work*, link into [DESIGN.md](DESIGN.md). For *what changed*, see [Tracker.md](Tracker.md).

## What this is

Lichborne is a DragonRealms (text MUD) game client. Electron + React + TypeScript. Connects via Lich5 (recommended) or direct SGE login. Windows x64 only. Single-developer project with a small, reachable tester pool.

**Product position (DESIGN.md §1, §24):** **display and configuration layer over Lich**, not a Lich-replacement. Lich owns automation (scripts, variables, triggers, repository). Lichborne owns rendering, themes, the panel system, profile portability, and surfacing Lich state. **Push back on requests that head toward duplicating Lich functionality** — say so before implementing.

## Principles (rules of the road)

1. **YAML is truth for user state. localStorage is a working copy.** New per-character settings need only a `scopedKey(character, ...)` write + `saveProfile()` call. Shared settings live in `_shared.yaml` and need typed entries in `SharedProfile` + `buildSharedProfile` + `importSharedProfile`.
2. **Lich is optional but recommended.** Every feature must work without Lich; Lich integration is additive.
3. **No silent data loss.** If a file can't be processed (future-version YAML, malformed XML), log it and **preserve it on disk**. Never overwrite content the code doesn't understand.
4. **Theme any new UI — and do a theme pass on every new feature.** All colors come from CSS custom properties in `themes.ts`; use `color-mix(in srgb, var(--x) N%, transparent)` for tints. When you add UI, audit its CSS: every surface / text / border color must be a `var(--…)`, and **eyeball it on a light theme** (Classic Light, Ivory, Mist, Parchment) — the `--bg-raised/sunken/base/hover` scale is spread out on dark themes but compressed near-white on light ones, so a literal that looks fine in the default dark theme can vanish on a light one (this is the single most-recurring bug class here: pitfalls #34, #55; B108 / B112 / B140 / B141 were ALL this). **The only color literals that are acceptable** (do NOT try to theme these): dark **drop-shadows** under menus/popovers (`rgba(0,0,0,.5)` — dark shadows read fine on light themes too); **text-halos over a *saturated* fill** (white text + `rgba(0,0,0,.9)` stroke on vital / exp bars — works on every theme precisely because the fill is always saturated, e.g. `.vital-text`, `.exp-rxp-bar-text`); and `var(--x, #fallback)` **fallbacks** (the fallback never fires since `darkBase` is the floor every theme merges over). Anything else hardcoded is a light-theme bug waiting to happen — let the cascade do the work. The v0.10.0 top-chrome features (app-bar, More menu, compact vitals, active-glow) were built/audited to this bar; the native Electron menu + About dialog are the deliberate exceptions (Win32-native, not themeable).
5. **Stream IDs preserve case end-to-end.** A Lich script that pushes to `LichScripts` lands in a panel called `LichScripts`. The `NEVER_DISCOVER` filter is the only case-insensitive check.
6. **Per-session everything.** Every IPC payload carries `sessionId`; every listener filters by it; every per-character state lives under `lichborne.{character}.{suffix}`. Cross-session bleed is the most common bug class.
7. **Fix root causes, not symptoms.** Past root-cause fixes (B14, B17, B20, B33, B36, B53, etc.) left inline comments explaining the current shape of the code. Read them before patching nearby logic.
8. **Ship reversibly.** Bold changes welcome (v0.6.3 maps rewrite, login redesign). Pair them with migration paths (profile schema versioning, NSIS rescue hook) so a bad rev never costs a tester their data.
9. **Wire every UI-manipulating setting to every surface it should affect — defining it isn't enough, and "new feature" means "make the settings reach it."** This is as load-bearing as theming (Principle #4): a setting that silently doesn't apply to a new surface is a bug. **Font size / line height:** game-area UI scales ONLY if it uses `var(--game-font-size)` / `var(--game-line-height)` — directly, or via `em` inheriting from a container anchored to those vars. **`rem` does NOT scale** — it's anchored to the html root (16px), not the per-character setting; never use `rem` for game-area text (pitfall #45). It's exactly why the icon bar, vitals bar, and Mode button silently stopped tracking the font setting. When you add a game-area strip/panel/button, **anchor its container to `font-size: var(--game-font-size)` and size children (and their heights/paddings) in `em`** so the whole thing scales without clipping — copy how `.panel-frame-body` ([panel-frame.css](src/renderer/styles/panel-frame.css)) and the panel roots in [panels.css](src/renderer/styles/panels.css) do it. **Accessibility:** high-contrast, color-blind, large-print, and epilepsy-safe are applied by `applySettingsToDOM` (re-applied after every theme write via the hook — pitfall #33); new UI inherits them only if it draws from the theme/overlay vars. **The check (do it for every new surface):** toggle each relevant setting — font size, line height, high contrast, color blind, large print, epilepsy-safe — and confirm the surface actually responds. App-chrome modals MAY deliberately opt out of game-font scaling (they have their own sizing), but that's a conscious call, not an accident.

## Architecture (pointers, not duplication)

### Process split
- **`src/main/`** — Electron main (Node). Sockets, XML parsing, Lich child process, file I/O. Built with esbuild (`build-main.mjs`).
- **`src/renderer/`** — React UI. Built with Vite. Talks to main only via `window.api.*` (preload.ts contextBridge).
- **`src/shared/types.ts`** — single source of truth for IPC payloads and game-event shapes. Imported by both sides.

### Multi-session model (DESIGN.md §13)
N character sessions across one OR MORE windows (v0.11.0). `src/main/main.ts` holds `sessions = Map<SessionId, Session>`; each Session owns its own `ConnectionManager`, `StormFrontParser`, `LichBridge`. Every push channel (`game-event`, `connection-status`, `raw-xml`, `error`) carries `sessionId`. **When adding any IPC, include `sessionId` in the payload.**

### Multi-window & decouple (v0.11.0)
A tabbed character can be **decoupled into its own OS window** while everything stays in ONE process — so cross-character features (Quick Send) keep working, Lich-launch stays coordinated, and profiles/localStorage aren't raced by a second exe. The model:

- **Main owns the session ROSTER; windows are views.** Pre-v0.11.0 `SessionsContext` ([SessionsContext.tsx](src/renderer/SessionsContext.tsx)) was the renderer-local source of truth for the session list. Now main is authoritative: each `Session` carries `meta` (characterId/account/character/game/useLich) + `ownerWindowId`, and `broadcastRoster()` pushes a `RosterEntry[]` (`session-roster` channel) to EVERY window on any change. [RosterContext.tsx](src/renderer/RosterContext.tsx) mirrors it; `useRoster()` exposes `roster` (all windows' sessions, for Quick Send), `windowId`, `isPrimary`, `myRoster` (owned). `SessionsContext` still drives this window's *own* tabs (rich `SessionStatus` stays local); the **decouple-sync effect in AppShell** ([App.tsx](src/renderer/App.tsx)) keeps it aligned: PULL owned sessions on mount (`get-owned-sessions`) + react to PUSH `session-acquire`/`session-release`.
- **Window registry + owner-targeted routing.** `windows: Map<webContentsId, BrowserWindow>` + `primaryWindowId`. Per-session output routes via `ownerWindow(s)` (falls back to primary if the owner is gone); app-global output goes to primary (auto-update), focused (menu), or sender (flash). `ownerWindowId` is the calling window's `event.sender.id`, set in the LOGIN handler.
- **Decouple = move ownership, NEVER touch the socket.** `session:move-window(sessionId, 'new' | 'main' | windowId)` reassigns `ownerWindowId`; the source window drops the tab (`session-release` → local `removeSession`) and the target mounts a fresh GameWindow. The socket/parser/LichBridge are untouched (verified: GameWindow unmount cleanup doesn't disconnect — pitfall #59). Three entry points: right-click tab, Window menu, Bulk Connect "separate windows" checkbox (persisted in `_shared.yaml` as `bulkConnectSeparateWindows`, default off). **Last-character guard** at 3 levels (main backstop + greyed Window-menu item + disabled right-click) — can't empty a window.
- **Lifecycle.** Closing a **secondary** window = graceful **logout** of its character(s) (`runSecondaryWindowClose`); **re-home** is explicit via Window → "Move Character to Main Window" (auto-closes the emptied window, no disconnect). Closing the **primary** = app quit (`runAppShutdown` flushes EVERY window's debounced profile saves, then drains all sessions). **Multiple exe instances are deliberately NOT gated** (isolated character sets); the shared-`userData` hardening is a separate deferred item.
- **State replay (seamless takeover) — see pitfall #60.** A window taking over a session gets a render-only replay (scrollback + every vital/room/etc.), gated so it never re-fires triggers or double-logs.
- **Cross-window state — three rules.** (1) **App-wide settings applied to the DOM** (the theme) don't auto-cross windows — each window has its own document. The fix is a `storage`-event listener in App that re-applies on change (`lichborne.theme`/`myThemes` → `initTheme()`); the window that made the change applied it directly and gets no storage event (no double-apply). App-wide settings that are READ FRESH per use (session-log settings, advanced/Lich settings) need NO sync. Per-character settings live in one window (the owner), so no cross-window race. (2) **Anything app-level that acts on "the active character in another window"** must route through main to that window: Profile Transfer remounts a moved-in target via `session:reload` → owner window (pitfall #56); the modal's active-target detection uses the **roster** (all windows), not the local `sessions`, or a cross-window target is mis-treated as inactive and its owner window overwrites the import. (3) **The native menu is one global menu for the focused window** — `refreshMenuState()` scopes Next/Prev/Close/Move enabled-state to the FOCUSED window's owned-session count (re-run on `win.on('focus')`), and `sendMenuAction` routes to the focused window. Don't gate Window-menu items on global counts.

### Lich launch & connect (v0.7.0; GTK-friendly spawn v0.9.x)
`LichConnection.launch()` spawns `rubyw lich.rbw --stormfront --dragonrealms` as a **detached** child (not a service). Lich takes its force-mode path: bind `127.0.0.1:11024`, then `accept` **exactly one** front-end and **close the listener** — one Lich = one session. Readiness is **not** a fixed delay: `connectWithRetry()` retries the real connection (250ms cadence, ≥30s cap) until the port accepts — the first success IS the session socket (no throwaway probes). `launch()` resolves on the child's `spawn` event and watches `exit` so a Lich that dies on startup fails fast (with its launch-log tail). Because each Lich serves one front-end then frees 11024, multi-character logins **must serialize** their spawn→connect window — `serializeLichLaunch()` in [ConnectionManager.ts](src/main/connection/ConnectionManager.ts) is a module-level promise chain shared across all per-session managers. SGE/eaccess auth runs outside that chain so it overlaps the queue wait. A failed launch `killProcess()`es its Lich so it can't squat the port.

**Spawn shape is GTK-friendly (v0.9.x).** Lich runs as a normal Windows GUI-subsystem process so Ruby/GTK scripts get a proper process/desktop context — the community-standard shape (matches how Frostbite / Genie launch Lich). Three things make it so, all in `launch()`: (1) it runs **`rubyw.exe`** (GUI subsystem), not `ruby.exe` (console) — `resolveRubyw()` derives `rubyw.exe` from the configured ruby path (`/ruby\.exe$/i` → `rubyw.exe`), falling back to the given path if the derived file doesn't exist (so a path already pointing at `rubyw.exe` is used verbatim); (2) **no `windowsHide`** (rubyw is windowless by nature — nothing to hide); (3) **no stderr pipe** — Lich's stdout+stderr are redirected to a per-character log file at `{userData}/Logs/lich-launch/{Character}.log` (truncated per launch), and `describeExit()` reads that file's tail for the error banner instead of an in-memory pipe. The redirected-console **pipe** of the old shape was the prime suspect for GTK flakiness (the kind of non-GUI process context that made widget pumps racy), which is why diagnostics moved to a file. `detached` + `unref()` + the child handle are kept (Lich is the proxy, must outlive us; the handle lets `exitInfo` fail a dead launch fast). The settings/default ruby path stays `ruby.exe` ([lichSettings.ts](src/renderer/lichSettings.ts)) — derivation handles rubyw at launch time, no migration.

**Ruby GTK script windows — product stance REVERSED (v0.9.x), verification pending.** The prior decision was "do NOT support GTK" (2026-05-30/06-01), on the theory that Lichborne's launch shape made GTK irredeemably flaky. **That decision is reversed** (Sekmeht, "I need to revisit the connection stuff to allow for GTK — it's the only way"). The v0.9.x **GTK-friendly spawn shape** (above: `rubyw.exe`, no `windowsHide`, no stderr pipe) was changed specifically to give Ruby/GTK a normal GUI-subsystem process context — the same shape Frostbite/Genie use, under which the same scripts work. **Status: expected to fix the GTK flakiness, pending empirical verification** (start `;vars setup` / `kill-counter` and confirm the window paints + survives interaction). Until that's confirmed in testing, treat GTK support as "should work now" rather than "guaranteed."

The historical root-cause analysis (still useful background): GTK scripts (`Gtk.queue { ... }` + `Gtk::Window.new`) showed (a) windows that failed to paint, (b) windows that painted only **intermittently** (kill-counter often surfaced only after a start→kill→restart cycle — timing-dependent, not categorically "never paints"), and (c) crashes on widget interaction (the `;vars setup` shape — a window that *did* paint could still take Lich down on interaction). The loop itself WAS running (`lich.rbw:135` calls `Gtk.main` unconditionally; `gtk.rb:148-155` yields via `GLib::Idle.add`), so the failure was a subtler process-context problem — which the old console-subsystem (`ruby.exe`) + hidden-window + redirected-stderr-pipe spawn produced, and which the new GUI-subsystem (`rubyw.exe`) plain spawn removes. **When a tester reports a GTK script not working:** first confirm they're on a v0.9.x+ build (the new spawn); if it still misbehaves, that's now a real bug worth investigating (capture the launch log at `{userData}/Logs/lich-launch/`), not expected behaviour. The `<streamWindow>` / `<pushStream>` XML approach (see `newkill-counter.lic`) remains the more portable recommendation for script authors (works in every front-end with no GTK runtime), and Lichborne-native equivalents are still preferred where they exist (`;vars setup` → the Lich Dashboard → Variables editor, v0.9.0). **The v0.9.0 proactive `--- Lichborne:` GTK advisory (the GTK detector + `client-notice` channel) was REMOVED in v0.9.x** alongside this launch change — it assumed GTK is broken, which is no longer the operating assumption (see the #54 tombstone). Pitfall #52 (the `;vars setup` GTK root-cause analysis) is kept as useful background. Pitfall #53 (write-through-Lich editor pattern) is unaffected.

### Game-text pipeline
`SGEConnection`/`LichConnection` → `ConnectionManager` (line events) → `LichBridge.interceptLine` → `StormFrontParser.parse` (stateful XML tokenizer) → `GameEvent[]` → batched IPC → `GameWindow`.

`StormFrontParser` is **stateful** (boldDepth, color stack, monoMode, injuryBuf, etc.). Call `parser.reset()` on new login. `<prompt>` is a good boundary for transient state.

Main text rendering uses **react-virtuoso**. Scroll pinning is a delicate state machine: `pinnedRef` snapshots before commit, `suppressUntilRef` gates programmatic scrolls (200ms; `scrollToBottom` 300ms), `onWheel` un-pins synchronously. Scroll-to-bottom is **instant** (`followOutput` returns `'auto'`). A story-window smooth-scroll feature existed v0.6.8–v0.6.11 (opt-in setting + flood-adaptive fallback) and was **removed in v0.6.12** — marginal, and a magnet for false bug reports. **Don't add `useEffect`-driven scroll logic** (or re-add smooth scroll) without reading GameWindow.tsx and DESIGN.md §23. Room-state updates funnel through the rAF pump (`roomQueueRef`), not direct `setRoomState` — see DESIGN.md §23 and pitfall #20.

### Profile system (DESIGN.md §20)
YAML files in `app.getPath('userData')/profiles/` — `%APPDATA%\lichborne\profiles\` in production (lowercase — `app.getName()` reads the package.json `name` field, NOT `build.productName`). `ensureProfilesDir()` in [profiles.ts](src/main/profiles.ts) is the canonical accessor. Two-stage migration (NSIS `preInit` + runtime fallback) rescues pre-v0.6.4 install-dir profiles.

- **Per-character setting** — write to `scopedKey(character, 'myThing')` and call `saveProfile()`. The dynamic `state:` pipeline picks it up automatically. No type changes needed.
- **Shared setting** — must be added to `SharedProfile` type + `buildSharedProfile` + `importSharedProfile`.
- **Schema versioning** — both files declare `profileVersion`. Migrations registered in `profile-migrations.ts` by SOURCE version. Adding optional fields is NOT a breaking change. Future-version YAMLs are skipped (preserved on disk).
- **Backups** — timestamped `{name}.yaml.{ISO-ish}.bak`, rolling retention of 5.

### Profile Transfer (F38, v0.10.0)
Platform-wide character setup Export/Import. As of v0.10.0 this is the **single Lichborne↔Lichborne path** — it superseded the old Automations export (that Export button was removed, and the ImportWizard's "Lichborne" source card was removed; the Automations panel's Import button is now legacy-client only, "Import from another client…" — see the Import wizard section). Launcher top-bar **"Transfer"** button → one modal, Export / Import tabs ([ProfileTransferModal.tsx](src/renderer/components/ProfileTransferModal.tsx), core logic in [profileTransfer.ts](src/renderer/profileTransfer.ts)). Export picks ANY source character (reads its YAML `state`, works disconnected) and writes a `.lb.yaml` bundle into a new **`Exports/` folder** (sibling of `profiles/`, `getExportsDir`/`ensureExportsDir` in [profiles.ts](src/main/profiles.ts)). Import lists files from `Exports/` (+ Browse… dialog defaulting there) and fans out to multiple already-added characters via checkboxes. Files: `profile-transfer:*` IPC in [main.ts](src/main/main.ts).

- **What transfers is an explicit allowlist** — `TRANSFER_CATEGORIES` in [profileTransfer.ts](src/renderer/profileTransfer.ts). Ten categories grouping exact `state` suffixes: Display & Accessibility (`settings` minus `panelFontSizes`), Panel Layout (zone added-flags / tabs / activeIds / heights / `panelWidth` / `panelFontSizes`), Panel View Preferences (`mapViewMode`, `lichMapScale`, `streamTimestamps`, `scriptPalette`, exp prefs), Theme (top-level `theme` + shared custom theme), and Highlights / Triggers / Macros / Aliases / Groups & Modes / Contacts. **Deliberately excluded**: `seededRepeatMacros` / `mainTopMigrated` (internal), `discoveredStreams` (ephemeral), `activeGroupStates` / `activeModeId` (only on Replace). **When you add a new per-character setting, decide if it belongs in a category here and add it** — settings default to NOT transferred until listed (safe default; a forgotten key is a papercut, never a target-breaker).
- **Non-destructive by construction** — import writes ONLY into the target's `state` map (per-key) + top-level `theme`. Identity/launcher top-level fields (`account`, `character`, `game`, `useLich`, `hidden`, `favorite`, `guild`, `circle`, `notes`, `profileVersion`) are never in `state`, so they're structurally unreachable.
- **Active vs inactive targets** — see pitfall #56. Inactive → YAML read-merge-write. Active (focused OR backgrounded session) → write localStorage working copy + remount the session via App's per-session reload nonce. **LB** is the project's short branding abbreviation (used in the `.lb.yaml` extension).

### Map system (DESIGN.md §19)
Two views: **Lich Map** (Lich image tiles) and **Genie Maps** (Genie XML-driven graph). Genie XML is the spatial source of truth for the graph view — coordinates come directly from the hand-curated XML files; no auto-layout, no BFS placement, no zone stitching. One zone visible at a time, picked by the dropdown or auto-switched when the player's room title matches a room in another loaded zone.

**Two prior approaches have been deleted; do not reintroduce.** The per-zone Genie Graph (`MapGraphView.tsx`, deleted v0.6.3) tried to render zones individually with the old augmentation matcher. The Lich-native BFS view (`LichGraphView.tsx` + `lichLayout.ts`, deleted v0.6.6) tried to make every Lich room renderable by inferring layout from `wayto` cardinals — but Genie's hand-curated coords and Lich's directional walks disagree in dense clusters, producing "type west, marker goes north" misrenders.

**Rendering matches Genie's `MapForm.cs` exactly.** 8×8 node rects are CENTERED on the XML position because Genie's `ConvertPoint(pos, 4)` SUBTRACTS the offset (MapForm.cs:187–193). Arc endpoints go through the XML position directly. Labels at `(pos + 1, pos + 1)` matching `DrawString(r.X + 1, r.Y + 1)`. Getting any of this wrong visibly misaligns label clusters; verify against the reference source instead of guessing.

**Arcs render in two passes** for dense-cluster legibility: under-pass at full opacity (drawn before nodes; lines disappear into rect fills inside clusters) + over-pass at 0.35 opacity drawn on top (faint trace across rects so you can follow a line all the way to its endpoint). Each pass collapses N arcs into 3 `<path>` elements (one per cardinal/climb/go category) via concatenated `M x,y L x,y` segments; same data, render twice, ~6 SVG elements total.

**Cross-zone stubs** are 1-room boundary markers with `note="MapXX_Name.xml"`. `isStubNode(n)` detects `.xml` in note aliases. Title lookup MUST prefer non-stub matches (otherwise the marker in zone A wins over the real room in zone B) — and it merges `byTitle ∪ byNormalized` *before* the non-stub filter, so a `byTitle`-only stub can't trap the match when the real room is reachable through the normalized form (B78).

**Ambiguous-title disambiguation in `currentLocation` (v0.7.0; substring desc added v0.11.2).** Same-titled rooms (the Crossing has 7 "Moonstone Street" rooms) resolve in this order: (1) single candidate → return it; (2) exact normalized **description** match; (2b, v0.11.2/B148) **substring** description containment — exactly-one-candidate, both strings ≥24 chars — because Genie's stored descriptions are usually a *truncation* of the live look so exact equality alone missed real rooms (the "stuck until a LOOK / telescope / combat re-emit" symptom); (3) **graph adjacency** — the candidate joined by a Genie arc to `prevLocRef` (the last resolved room); (3b) cross-zone stub adjacency; (4) conservative cross-zone hold (keep `prev`, 3-strike escape, B110); (5) file order (`pool[0]`) last resort. Adjacency is what makes the marker track *while running* — the game streams room titles with no fresh description per step. The Lich Map's `findRoom` ([mapTypes.ts](src/renderer/components/panels/mapTypes.ts)) carries the SAME exact-then-substring desc logic (B150) so both views behave identically on the title+desc fallback path. `prevLocRef` is updated one step behind by an effect, advanced only on a non-null match. **The two maps differ in their PRIMARY signal:** the Lich Map matches by **room id first** (`lichDb.get(roomId)`, exact/robust — and now that the parser reads the Simu room-id-flag `[Title] (id)` format, pitfall #65, it tracks reliably for flag-on players) and only falls to title+desc when the id is absent; the Genie Map has **no room ids in its data**, so it is *always* title+desc and inherently degrades through ambiguous rooms DR didn't describe — that's a fundamental limit, not a bug. Matching is per-session/state-driven (MapPanel gets each GameWindow's own `roomState.{title,desc,roomId}`, gated on the per-character `showMapOverlay`, NOT `isActive`) so it tracks for active AND backgrounded characters; only the camera/layout is active-gated (pitfall #24).

**Click model (v0.6.8): left-click pins, right-click walks.** Left-click a regular room → toggles a pinned gold BFS path (no walking). Left-click a stub → switches the displayed zone to its target XML. Right-click anything → walks (BFS in-zone, `sendWalkPath`). The map only auto-switches zones when the player's `roomTitle` actually lands in the new zone — never on walk completion (the game can block any command; racing the map ahead strands the user).

**Camera follow-the-player** uses `useLayoutEffect`, not `useEffect`, so the transform update lands in the same paint frame as the indicator's new world position. Default ON; manual pan/zoom turns it off; the ◆ button re-enables. Always-center model (not margin-snap). The follow effect gates on `followNode` — the *same* `visibleById.get(...)` lookup the indicator uses — so "marker visible" and "camera following" can't desync.

**Smooth camera motion.** Pan group is positioned with the CSS `transform` *property* (not the SVG attribute) so `.genie-pan-smooth` can `transition` it (`150ms linear` — linear, not ease-out, because a follow camera re-targets every step and ease-out's velocity reset pulses visibly). `snapTransform` (delta > 600px or > 20% scale vs last-painted transform) drops the transition for big jumps so zone switches / ◆ / fit cut instantly. The indicator carries the *same* transition + snap flag so it stays screen-centred during walks instead of bouncing. Gated on the `mapAnimations` setting (default ON): pan group + indicator only get `.genie-pan-smooth` when `!isDragging && mapAnimations`. (v0.6.8–v0.6.11 gated this on a separate `smoothScroll` setting; v0.6.12 removed that and folded the map glide under `mapAnimations`.)

**Current/selected/hover/hover-path/pinned-path indicators are hoisted OUT of `nodeRects`** as single overlay elements. Pre-hoist, walking re-built the entire `nodeRects` JSX every step. The current-room indicator is a sonar locator: two `genie-here-ping` expanding rings + the solid `INDICATOR_R` ring; the pings are exempt from the animation-pause via a higher-specificity rule.

**Per-color effect system (v0.6.7).** Every COLOR_LEGEND category has its own visual signature (heartbeat for Healer, coin glint for Shop, ripples for Water, leaves for Lumberjacking, dirt for Mining + Trailhead, vortex/drift/rise sparkles for the magical 4, etc.). The structure is uniform — adding a new effect is a 4-step pattern: (1) declare a `*_COLORS` set or extend `MOTE_EFFECTS`, (2) add a memo at the appropriate place in the component that builds the SVG elements per visible-node match, (3) add a CSS keyframe in `map-panel.css`, (4) render the memo at the correct layer in the SVG tree. Stub `↗` glyph always wins over category glyphs (cross-zone identity > resource category). `getMoteContrastColor(hex)` picks mote fills based on room luminance (light bg → dark motes, dark bg → light motes). `normalizeNodeColor()` in `parseGenieZone` converts CSS color names (`"Blue"`, `"Red"`, etc.) to canonical hex so downstream effect lookups don't silently miss rooms.

**Animation pause — two freeze classes (v0.6.7, extended v0.6.9).** The pan group `<g>` carries ONE of two mutually-exclusive freeze classes, both cascading `animation-play-state: paused` to descendants:
- `genie-pan-dragging` — *transient* freeze when `isDragging || inMotion`. `isDragging` flips on mousedown/up; `inMotion` flips true on any `currentLocation` change and resets a `MOTION_QUIET_MS = 800` timer. The locator sonar ping is **exempt** (`.genie-pan-dragging .genie-here-ping` keeps it running) — you still want to see yourself mid-walk.
- `genie-anim-off` — *permanent* freeze when `settings.mapAnimations` is off (default ON). **No ping exemption** — "off" stops the ping too.
The className picks one or the other (`!mapAnimations ? 'genie-anim-off' : (isDragging||inMotion) ? 'genie-pan-dragging' : ''`) so the ping-pause and ping-exemption rules never collide.

**Category effects are OMITTED (not paused) and CAPPED (v0.7.0).** Two layers of defense, both because SVG CSS animations are expensive — each animated element forces Recalculate Style every frame and is a compositor-layer candidate:
- *Omitted during motion.* `animation-play-state: paused` stops an animation advancing but leaves the element layer-promoted — profiling during travel showed residual Layerize ~21% even with effects "frozen". So the 10 animated per-category effect groups (sparkles, heartbeats, coin glints, ripples, bubbles, caution/implode rings, leaf/dirt falls, XP rises) render only when `showEffects = mapAnimations && !isDragging && !inMotion`. During travel/pan/off they're not in the DOM at all. They re-mount once the player has been still for `MOTION_QUIET_MS` (1500ms — long enough that normal sub-second running cadence never re-mounts mid-run; a short window would tear them up and down every step).
- *Capped to the nearest N rooms.* Idle in a dense zone (the Crossing) all effects on was ~29% Recalculate Style. The effect memos iterate `nearbyNodes`, not `visibleNodes` — `EFFECT_CAP` (30) rooms nearest the player (or zone-centre when viewing a zone you're not in), or the stable `EMPTY_NODES` ref when `!showEffects`. So at most ~30 rooms animate regardless of zone density, and the effect memos don't recompute during travel (dep is the unchanging `EMPTY_NODES`). **`auras` deliberately still uses `visibleNodes`** — they're static (no animation, no per-frame cost) and they ARE the colour map.

The freeze classes still cover what *stays* rendered (the sonar ping, the static/fire auras). Node **hover** is likewise suppressed during motion (`onNodeHoverEnter` checks `inMotionRef`) — the map scrolling under a stationary cursor otherwise fires a storm of pointer enter/leave → re-renders.

**`mapAnimations` setting (v0.6.9; sole map-motion switch as of v0.6.12).** `mapAnimations` (default ON) is the single toggle for ALL Genie map motion — the per-color category effects AND the camera glide. Off → `genie-anim-off` freezes effects and the glide is disabled (camera snaps). Per-character `AppSettings` field, persists via the dynamic profile `state:` pipeline, threads `GameWindow → PanelFrame → MapPanel → GenieMapView` (the `autoLinkUrls` path). (v0.6.9 also had a `smoothScroll` setting; removed v0.6.12 with the story-window smooth scroll.)

**Genie parse cache (v0.6.7).** First-time parse of a 122-XML maps folder takes several seconds (DOMParser is synchronous and chunky). `genie-cache:load` / `genie-cache:save` IPC handlers in `main.ts` serialize the parsed `Map<zoneId, GenieZone>` to `userData/genie-cache.json`. Subsequent launches verify a fingerprint (sorted `filename:mtimeMs:size`) against current folder state and skip re-parsing if it matches — load drops from seconds to ~50ms. Version field (`GENIE_CACHE_VERSION`) lets us invalidate every user's cache when the GenieZone shape changes without manual cleanup. Cache write is fire-and-forget after parse.

### Session Log (DESIGN.md §28, shipped v0.7.0)
Per-character daily plain-text logs. **Capture** happens in `GameWindow.onGameEvent` → `logToSession()` → `session-log:append` IPC → buffered per-character writer in [sessionLog.ts](src/main/sessionLog.ts). Files: `{userData}/Logs/{Character}/{Character}_YYYY-MM-DD.log`, line format `[HH:MM:SS][stream] text` (the date is in the filename; parsers also accept the legacy `[YYYY-MM-DD HH:MM:SS.mmm]` form). - **Config is app-wide, NOT per-character.** All Session Log preferences (capture gates, retention, compression, raw-size cap, the Recent-tail filter, the Export-builder format prefs) live in one `SessionLogSettings` object — [sessionLogSettings.ts](src/renderer/sessionLogSettings.ts), stored in `_shared.yaml` via `SharedProfile.sessionLog`. `logToSession` calls `loadSessionLogSettings()` fresh per batch (it's a tiny localStorage read — no stale-closure ref needed, and a Settings change applies instantly to every open character). - **Size management (v0.7.0):** closed (non-today) day-files are gzip-compressed to `.log.gz` (~85-90% smaller) by background `maintain()` — runs at session start + day-rollover, gated by `sessionLogCompress`. All readers route through `readLogFile()` (gunzips transparently) and `resolveDayFile()`. Two retention limits: `pruneOldLogs` (age, `.log` + `.log.gz`) and `enforceRawSizeCap` (caps *uncompressed* `.log` bytes only — never `.gz`, never today's file). If you change the line format or add a maintenance limit, keep both parsers (`LINE_RE` in the modal, `EXPORT_LINE_RE` in main) accepting the legacy form.

- **Read IPC:** `session-log:list-days`, `read-day` (tail + `beforeLine` pagination), `list-streams` (single day, or a range when `toDate` is passed), `search` (returns `SessionLogSearchHit` with `lineNo`/`total` for jump-to-tail), `build-export`, `disk-usage`, `open-folder`. ## Pitfalls (read before touching adjacent code)

> **Adding a new pitfall — read this first.** Pitfalls are numbered sequentially and the numbers ARE referenced from other comments / commit messages / docs (e.g. "see CLAUDE.md pitfall #33"). When adding a new one, **find the highest-numbered existing pitfall and add the new one AFTER it, with the next number** — do NOT insert before an existing pitfall or renumber an existing one. The natural editing instinct (using Edit with an existing pitfall as the anchor `old_string` and putting the new content first in `new_string`) puts the new pitfall in the wrong position; the correct anchor is the `---` separator at the END of the pitfalls list, with the new pitfall inserted between the last existing pitfall and that separator. Sekmeht caught me making this exact mistake three times across v0.8.7 and v0.8.8 (B121, B124, B126) — each time renumbering an older pitfall instead of just appending. The fix path when caught: revert the renumbered older pitfall to its original number, then append the new one at the end. The numbering is `1, 2, 3, ..., 31, 33, ...` (32 has been missing for ages; don't try to fix the gap, it's referenced from elsewhere by its current number).

1. **`app.getName()` returns `name`, not `productName`.** userData is `%APPDATA%\lichborne\` (lowercase). The `productName: "Lichborne"` under `build:` only affects installer / Start menu display. Mixing the cases caused a v0.6.4 hotfix to silently no-op.
2. **electron-builder NSIS `customInit` runs AFTER the previous uninstaller.** Use `preInit` and read the previous install location from `HKCU\Software\${UNINSTALL_APP_KEY}` (`$INSTDIR` isn't set that early).
3. **NEVER put user data inside the install directory.** NSIS `RMDir /r $INSTDIR` on every upgrade. Pre-v0.6.4 lost everyone's profiles this way.
4. **`StormFrontParser` is stateful and lives for the session lifetime.** Reset on new login. Stream-mode state can bleed across `<prompt>` boundaries.
5. **Scroll pinning fights you if you add naive useEffect logic.** Read GameWindow.tsx + DESIGN.md §23 before adding any programmatic scroll.
6. **Stream IDs are case-sensitive everywhere except `NEVER_DISCOVER`.** Lowercasing them anywhere will break Lich scripts. See B34.
7. **`lichPort` is the Lich front-end port, not an SGE shard port.** Don't look it up from a per-game table at credential build-time (v0.6.3 bug).
8. **`setState` in a mousemove handler must be gated on a hover sentinel** or you re-render on every pixel. See `GenieMapView.tsx` `onMouseMove` (gates on `dragRef.current`).
9. **Genie node rects are CENTERED on the XML position, not top-left anchored.** `ConvertPoint(pos, 4 * scale)` SUBTRACTS the offset (MapForm.cs:187–193). Anchoring top-left shifts every node 4px down-right and misaligns label clusters. Verify against Genie source for any map-rendering math change.
10. **`pointer-events: none` on the SVG pan group breaks click-to-walk.** `isDragging` flips true on mousedown BEFORE click fires — toggling `pointer-events` on the parent at that point makes the click target the SVG root, not the inner node `<g>`. Gate hover at the React layer (check `dragRef.current` in the hover handler) instead.
11. **Use `useLayoutEffect` for camera-follow.** The map's follow-the-player effect must update the transform in the same paint frame as the indicator's new world position. Plain `useEffect` produces a 1-frame "indicator at new spot, camera still old" flash at high walk rates.
12. **`useRef(initialValue)` initialized to a meaningful prop is a footgun.** The "did this change since last render" pattern requires `useRef(null)` (or whatever sentinel ≠ the first real value) so the first non-null render is treated as a change. See `lastLocationRef` in GenieMapView for the cautionary tale.
13. **Genie XML colors can be CSS color names, not just hex.** Some maps use `color="Blue"`, `"Red"`, `"Aqua"`, `"Lime"`, `"White"`. SVG `fill` accepts both, but `COLOR_LEGEND` and every effect lookup keys by hex. `normalizeNodeColor()` in `parseGenieZone` converts these to canonical hex at parse time so downstream code sees a single form. If a new room mysteriously has no aura or animation, check whether its color attribute is a named color we don't normalize.
14. **`opacity` SVG attribute conflicts with CSS animation on `opacity`.** Per SVG2 spec CSS wins, but having both set creates ambiguity that browsers can handle inconsistently. When applying an animated class to an SVG element, omit the `opacity` attribute (set to `undefined`) so CSS owns it cleanly. See the aura render's `opacityAttr` ternary.
15. **`indicator.id` is case-normalized to lowercase by the parser** (`raw.replace(/^Icon/i, '').toLowerCase()`) for the typed-flag checks AND the generic event payload. Renderer keys (`indicators.poisoned`, `indicators.bleeding`, etc.) are lowercase. Don't switch on `IconBLEEDING` directly downstream — match the lowercase form.
16. **If you change the `GenieZone` / `GenieNode` / `GenieArc` shape, bump `GENIE_CACHE_VERSION`** in `main.ts`. The parse cache stores the entire serialized zone graph; a shape change without a version bump means old caches deserialize into wrongly-shaped objects and produce silent runtime errors deep in the renderer. The version bump invalidates every user's cache on next launch with no manual cleanup.
17. **Map animations pause via the parent `<g>`'s class, not per-element.** `.genie-pan-dragging *` cascades `animation-play-state: paused` to all descendants. Don't add `pointer-events: none` to that group when toggling drag/motion — it would shift the click target off the inner node `<g>` and silently break click-to-walk (the mousedown sets the class before click fires).
18. **The map pan group and the current-room indicator must transition in LOCKSTEP.** Both carry `.genie-pan-smooth` (same `150ms linear`) and share the `snapTransform` flag. The indicator lives inside the pan group, so its screen position is `pan ∘ indicator`; if only the pan transitions while the indicator's world position jumps instantly, the halo bounces off-centre every walk step. Matched transitions make the interpolations cancel. Don't change one easing/duration without the other.
19. **No inline `height: 100%` on the `GenieMapView` outer wrap.** It's a flex child (`.map-canvas-wrap` → `flex: 1; min-height: 0`). An inline `height: 100%` overrides flex sizing and, evaluated against the parent's full height before the parent subtracts its own toolbar, pushes the MapPanel's view-selector toolbar off-screen at narrow window heights. Keep it `display: flex; flex-direction: column; minHeight: 0` and let CSS own the height.
20. **Room-state updates go through the rAF pump, not direct `setRoomState`.** Fast running emits multiple `room-title` events per frame; React 18 auto-batches them so only the last room survives and the map skips rooms. `roomQueueRef` + a `requestAnimationFrame` loop apply one per frame (cap 8). Don't bypass it.
21. **`Home`/`End` are native (command-box cursor) when the command input is focused.** Only `Ctrl+Home`/`Ctrl+End` and the focus-elsewhere case scroll the story window. `PageUp`/`PageDown` always scroll. Testers expect text-editing keys to edit text (B77). Don't re-claim plain `Home`/`End` for scrolling.
22. **`LichBridge.interceptLine` suppression is poll-aware — don't make it match on output format alone.** The auto-poll's `;listall` response is consumed (hidden) so it doesn't spam the game window, but a player-typed `;list`/`;listall` produces the same format. `LichBridge.pollScriptList()` arms a 4s window; `interceptLine` consumes a matching line only while armed, otherwise lets it through (B79). Route the auto-poll through `pollScriptList()`, never `injector.pollScriptList()` directly.
23. **Never put CSS `contain` on `.text-line` / `.text-line-wrap` (the react-virtuoso rows).** Layout containment isolates each row from Virtuoso's own item-measurement / scroll-offset bookkeeping, so the scroll position can't be held stable when the list updates while the user is scrolled up — what they're reading gets dragged off-screen (B84). It was a v0.6.8 per-row perf hint, removed v0.6.12. The CSS carries DO-NOT-RE-ADD comments.
24. **An inactive character's GameWindow is `display:none` — its DOM measures 0×0.** Every inactive tab stays mounted and keeps processing game events (room titles, etc.), but `clientWidth`/`clientHeight`/`getBoundingClientRect` all read **0** until the tab is shown. Any layout-dependent logic must (a) guard against 0 dimensions so it doesn't write garbage (`centerOnCurrent`, `fitToView`, the follow `useLayoutEffect` all bail on `!w||!h`), and (b) re-run when the tab becomes visible again — GenieMapView uses a `ResizeObserver` on the SVG to recenter the camera on the 0→size transition. Without (b), a character that travels while its tab is hidden tabs back with the camera stranded rooms away. This is the canonical multi-character map pitfall: hidden ≠ unmounted, but hidden = unmeasurable.

36. **Contact tracking has TWO long-running effects (last-seen + time-spent polling) that BOTH share the same `pendingContactsRef` buffer — every read AND every state-change must respect it.** Established in B119 + F34. 37. **Encounter / presence detection needs TWO gates, not one — absence-threshold AND cooldown.** F34's `encounterCount` increments only when a contact is detected AND `(now - lastSeen) > ABSENCE_THRESHOLD_MS` AND `(now - lastEncounterAt) > ENCOUNTER_COOLDOWN_MS`. The absence-threshold (90s, > polling-tick 60s with a margin) gates "they actually LEFT since last seen" — without it, a contact who stays in your room for 10+ minutes ticks the counter every time `room.players` changes (someone else enters / leaves) because the last-encounter timestamp is old. The cooldown (10 min) gates "even if they left, don't double-count a quick exit-then-re-enter." `lastEncounterAt` updates only on actual increments; `lastSeen` updates on every detection. Both timestamps are needed and serve different purposes — don't fold them. When adding any new contact-state-transition feature (e.g. "first seen on day X" badges), think in terms of presence-edge detection rather than just "is this contact here right now," and use the same absence-threshold + cooldown shape so we don't end up with multiple half-correct definitions of "encountered."

43. **When the user can manually scroll past where the scroll API thinks the bottom is, the bug is in LAYOUT, not scroll math.** Established in B122 across v0.8.7 → v0.8.8. The half-row clip at font 13+ ate three failed hypotheses before landing on a real fix: (a) v0.8.7 lowered a `totalListHeightChanged` threshold from 2px → 0.5px — didn't help; (b) v0.8.8 first attempt replaced raw `el.scrollTop = el.scrollHeight - el.clientHeight` with `virtuosoRef.current?.scrollToIndex({ align: 'end' })` on the theory Virtuoso's internal accounting differed — didn't help; (c) Sekmeht's diagnostic — "I can manually scroll down ~half a row past where the snap lands" — proved BOTH APIs agreed on the wrong target. The fix that worked: a `components.Footer` on Virtuoso (`<div style={{ height: 14 }} />`, fixed 14px, gated to `settings.fontSize >= 13`) — defensive layout slack that doesn't require knowing why the scroll APIs land short. **General principle**: scroll-snap math problems are usually solvable by adding layout slack rather than by chasing what the math is doing. If you're tempted to write a third hypothesis-based fix to scroll math, stop — add invisible padding below the content instead. Specific to this case: the 14px footer is **gated on font ≥ 13** because at font 12 and below row heights round to clean integer pixels and no clip occurs; the slack would just be empty space below the last row (visible gap above the vitals strip). **Pair fix shipped in same release**: a separate `useEffect([settings.fontSize, settings.lineHeight, settings.largePrint])` re-snaps via `scrollToIndex({ align: 'end' })` ONLY when `pinnedRef.current` was already true — font / line-height / large-print changes reshape every row's height, growing scrollHeight, leaving the previously-pinned scrollTop short of the new bottom. Bail on `!pinnedRef.current` so scrolled-up readers aren't yanked back. rAF before the snap so Virtuoso has a frame to re-measure rows at the new dimensions. The threshold check in `totalListHeightChanged` is left at 0.5 (responsive) but doesn't carry the load — the footer does.

One `;eq` (not two separate commands) because two commands race — a standalone `Vars.save` could run before a separate `;vars delete` finished, persisting the pre-delete state and leaving the DB stale until the 5-min timer. `Vars.save` is mutex-guarded (vars.rb:78) and rewrites the whole blob, so a deleted key actually leaves storage and the SQLite read view is correct on the next refresh. `Vars[]=`/`Vars.save` are Lich's documented public API (as stable as the `;vars` command, and what `;vars` calls internally) — this is NOT "running arbitrary Ruby," it's the same API surface. **Escaping:** values/names are injected into Ruby single-quoted literals via `rubyLit(s)` (escape `\` and `'` only — those are the only specials in a Ruby `'...'` string) so values with quotes/spaces/backslashes survive. Mirror Lich's own coercions (vars.lic coerces literal `true`/`false` to booleans — so do we, for display parity). **Silent send:** the `;eq` runs via a no-echo path (`onRunCommand` wired to `window.api.sendCommand(sessionId, cmd)` directly, NOT the echoing `sendCommand`) so the raw Ruby doesn't spam the game window — the optimistic UI update is the user-facing feedback. **Read stays SQLite** (`lich:get-vars` + Marshal) — structured display of lists/hashes/times, cross-scope browse, graceful per-row parse-error degradation. The read/write asymmetry is deliberate and correct: read wants structure + any-scope browsing (the Marshal storage format is a far more stable contract than the cosmetic `;vars list` output, and command-read would need a stream-capture/suppress mechanism like the `;listall` poll); write must mutate live memory. **Two gates on edit controls**: (a) `session.useLich` AND (b) the selected scope equals the connected character's own scope (`${game}:${character}`) — `;eq Vars[...]` only touches the CONNECTED session's memory, so editing while viewing another character's scope would silently write the wrong character's vars. Viewing other scopes is read-only with a note. **Optimistic overlay** (`pendingRef`, keyed by scope → name → change) bridges the sub-second gap between the write and the next DB read so ↺ / reopen never flicker an edit back to its old value; it survives a refresh by re-applying over the DB read, and is discarded on modal close (the forced save makes the DB authoritative-fresh by then, so this is belt-and-suspenders). **When adding a new "edit Lich state" surface**: follow this exact shape — write through Lich's public Ruby API + force its save in one `;eq`, read through whatever structured path exists, gate on connected-own-scope, send silently, overlay optimistically. Don't reach for a direct DB writer or a Marshal serializer; the race with Lich's in-memory authority is the trap.

60. **A render-only "replay" reseeds a window taking over a session — gate it at the side-effect choke points, snapshot sticky state, and HOLD live delivery during the handoff.** Established by the multi-window state-transfer (v0.11.0), refined across three tester-found bugs. A freshly-mounted GameWindow starts blank; to make decouple/re-home seamless, main replays the session's recent history to the window that took it over. Three things make this correct, each from a real bug:
    - **(a) Gate side effects, don't reprocess them.** The replay batch carries `replay: true` ([GameEventBatch](src/shared/types.ts)). The renderer rebuilds display + game state from it but MUST NOT re-fire triggers (would re-send commands!), re-log the session, or re-count fires. All trigger firing funnels through `processLineRef`/`processVariableChangeRef` and logging through `logToSession`/`logHighlightFiresRef`, so gating those FOUR choke points on a `replayingRef` (set true at the batch top in `onGameEvent`, reset false at the bottom) covers every call site at once — far safer than threading a flag through the ~14 individual call sites. The room/vital STATE setters (setVitals etc.) are NOT gated (rebuilding state is the point).
    - **(b) Snapshot sticky state — a ring buffer alone loses unchanged vitals.** Main keeps per-session `stateSnapshot: Map<key, GameEvent>` (latest of each vital/indicator/RT/CT/stance/spell/hand/room/exp/injury/exits, via `snapshotKey`) PLUS a bounded `historyBuffer` (scrollback text). The first design was a single ring buffer of recent events — but a vital sitting at 100% hadn't emitted a `vital-update` recently, so it fell off the end and restored BLANK (only actively-changing bars survived). The snapshot keeps the latest value of every bar regardless of age; replay sends snapshot FIRST then history.
    - **(c) Only replay to a MOVED-into window, and HOLD live during the handoff.** Replaying on EVERY GameWindow mount doubled the login stream on a fresh connect (the replay overlapped the live stream still filling the buffer). Fix: a session only earns a replay when MOVED (`replayTarget` set on `session:move-window`, one-shot, checked in `session:request-replay`) — a fresh connect never gets one. AND because a session can still be actively streaming *during* a move (bulk-connect logs in then decouples), `holdingForReplay` stops live `game-event` sends to the window from the moment of the move until the replay is delivered (events accumulate into the buffer, the replay delivers them once, then live resumes) — otherwise the in-flight login lines show live AND in the replay. A 5s `scheduleReplayHoldRelease` safety net resumes live if the window never requests its replay. The buffer is filled at FLUSH time (events actually sent), kept disjoint from the pending `eventQueue`, so a replay can't double a not-yet-flushed event either. **When adding a new "rebuild a window from history" path, reuse this shape: gate side effects via a replay flag, snapshot sticky state separately from scrollback, and never let live + replay overlap.**

Hardcoding VS-Code-dark hex (`#9cdcfe` etc.) is the pitfall-#34 light-theme bug at scale — light-blue keys vanish on white. The fix is NOT to add 8 vars × 20 themes; it's to add the 8 `--syntax-key/string/number/literal/comment/meta/type/symbol` vars to `darkBase` ONLY, each as `color-mix(in srgb, <fixed-hue> N%, var(--text-primary))`. Because the nested `var(--text-primary)` resolves in each element's (theme's) context at use time, one definition yields a readable, hue-distinct value on every theme automatically. Consumers just say `color: var(--syntax-key)`. **When you need a palette of fixed-meaning hues that must stay legible across themes, define them once in `darkBase` as color-mix-toward-`--text-primary`, not per-theme.** (Status/semantic single colors follow the same idea via `color-mix(fixed-hue, var(--bg-base/--text-primary))` — pitfall #55.) Related v0.11.2 work: the per-theme `--text-dim`/`--text-faint` ramp was compressed toward `--text-muted` (the dim tail was below readable contrast on several surfaces), and `classicLight` had an inverted ramp (`dim`/`faint` = pure black, more prominent than body text) — a reminder that the 5-level text ramp must be monotonic toward the bg on light themes too.

64. **Per-window state delivered by PUSH-only IPC is race-prone for a freshly-opened window — pull it on mount too (B149, v0.11.2).** Multi-window state (the session roster) is broadcast from main on every change (`broadcastRoster` → `session-roster`), including on a window's `did-finish-load`. But a brand-new window's renderer registers its `onSessionRoster` listener only AFTER React mounts (a `useEffect`), so it LOSES the `did-finish-load` broadcast — and if no further change fires (a stable connected session emits no status events; `session:set-name` only broadcasts on a name *change*), the empty value never self-heals. Symptom: Quick Send (which renders nothing when the cross-window roster is empty) did nothing via Ctrl+Shift+Enter in a decoupled window — the hotkey fired, but `RosterContext.roster` was `[]`. `SessionsContext` never had this bug because it **pulls** owned sessions on mount (`get-owned-sessions`) in addition to reacting to push (`session-acquire/release`); `RosterContext` was left push-only — the oversight. **Fix shape (the rule):** every per-window context that mirrors main-owned state must **PULL the current value on mount AND subscribe to future pushes** — never push-only. Added `get-roster` (returns `buildRoster()`) called in RosterContext's mount effect next to the subscribe. When you add a new cross-window broadcast channel, add its pull-on-mount handler at the same time. (Pull-resolves-after-subscribe ordering is fine: a push arriving between subscribe and pull isn't lost, and a late pull just reasserts the same authoritative list.)

65. **DR carries the room id in the `<streamWindow>` subtitle in TWO formats — parse BOTH, and treat its absence as normal (B151, v0.11.2).** The subtitle (which also carries the room title) can be either `[Whistling Wood, Barrows - 9479]` (id INSIDE the brackets after ` - `) OR `[Arthelun Ruins, Courtyard] (56107)` (title in brackets, id in PARENS after them). The parens form is the StormFront-standard `[Title] (uid)` — it's DR's **Simutronics room-id flag** display, an account setting the player can toggle; `(**)` means "room has no id" (unmapped, e.g. `[The Heavens] (**)` during a Moon Mage telescope flip). Pre-v0.11.2 the parser ([StormFrontParser](src/main/parser/StormFrontParser.ts) streamWindow handler) only parsed the dash form, so flag-on players got `roomId: undefined` → the Lich Map's instant `lichDb.get(roomId)` lookup never fired (forcing the fragile title+desc path — the "location lost during movement" symptom) and `$roomid` was empty. Fix: when the dash form misses, also match `\]\s*\((\d+)\)` after the closing bracket. **Crucial constraint — the flag is OPTIONAL, so id-absence must stay a no-op:** when neither form is present (flag off) OR the id is `(**)`, `roomId` MUST remain `undefined` exactly as before, and the whole downstream pipeline must keep working id-free (MapPanel falls back to `findRoom` title+desc; `$roomid` renders empty; `triggerCtxRef.roomId` only updates when `evt.roomId != null`). Never make a parsed room id load-bearing — it's a bonus signal when present (and the Genie map can't use it at all; its data has no game room ids, pitfall in §19 / B148), never a requirement. The companion id source is `<nav rm='X'/>` (pitfall #46) — when present it agrees with the subtitle; the telescope case shows DR sometimes sends an EMPTY `<nav/>` and relies on the subtitle parens, which is exactly why both subtitle forms must be parsed.

66. **Text selection across a virtualized list loses unmounted rows on copy — keep a generous `increaseViewportBy` buffer (B152, v0.11.2).** The story window is react-virtuoso; it mounts only the visible rows plus a small overscan and UNMOUNTS off-screen rows. A browser text selection lives on DOM nodes — when a selected row unmounts (you scroll while drag-selecting), it silently leaves the selection. So the mouseup auto-copy (`getSelection().toString()` → `window.api.writeClipboard` in [GameWindow.tsx](src/renderer/components/GameWindow.tsx)) captures only the rows still mounted at release — "a portion from the end" (the final viewport + overscan). The pragmatic, low-risk fix is `increaseViewportBy={{ top: 3000, bottom: 3000 }}` on the Virtuoso so ~150 rows each side stay mounted, keeping several screenfuls of selection intact through the auto-scroll (rows are memoized + `lines` is capped, so the extra DOM is cheap; scroll math is untouched). **Known ceiling:** a selection scrolling >~3000px past its start still can't be fully captured by a DOM-selection copy — the real ceiling-free fix is to reconstruct the selected text from the `lines` DATA MODEL (capture the anchor line id + char offset at mousedown — before it can unmount — map the focus at mouseup, slice `lines[anchor..focus]`), which was deferred as too intricate for the delicate scroll path. **General rule:** any copy/select/Ctrl-A feature over a virtualized list must either (a) carry a large enough mounted buffer for the expected selection size, or (b) reconstruct from the data model — never trust `getSelection().toString()` to span the full logical list. (Don't "fix" this by un-virtualizing the list — that's the perf foundation of the whole text window.)

## Proactive bug-checking

The goal is to find bugs before testers do — both new bugs the current change might introduce, and **latent bugs from prior versions that are still living in the current code**. Lichborne is single-developer with a small tester pool; every bug we ship is one a tester has to find and report, costing them and us time. Two complementary practices.

### Iron rule: read the actual code, not memory or docs

Every claim you make about how the code behaves must come from reading the file you're claiming about, IN THIS SESSION. Not from CLAUDE.md, not from pitfalls, not from a Tracker.md entry describing the code, not from "I remember how this works." Documentation describes the code's intent and history; only the source describes what it currently does. The B114 false-positive that prompted this rule (v0.8.7 retroactive sweep): pitfall #33 described the theme-overlay re-apply hook pattern, I worked from that description, claimed a multi-character bug existed, and was wrong — the actual effect at [GameWindow.tsx:954-973](src/renderer/components/GameWindow.tsx#L954-L973) had `isActive` in its deps and re-applied overlays directly on tab switch, which the pitfall didn't mention because the pitfall was scoped to a different scenario. Acting on the false positive would have shipped a redundant fix and possibly introduced a real new bug. The fix-cost of a false positive is higher than the fix-cost of doing the read.

**Verification protocol** when claiming a bug exists or doesn't exist:
- Grep for the specific function / state / effect you're reasoning about.
- Read the relevant lines yourself (Read tool with line offsets, not just the grep snippet — grep context can mislead by hiding deps arrays, guard clauses, or cleanup returns).
- Confirm the deps array of any `useEffect` you reason about — pitfalls describe patterns, not necessarily the current deps list.
- Confirm the cleanup function — pitfalls describe what's supposed to happen, not necessarily what does.
- If a pitfall and the current code disagree, **the code wins** — and the pitfall needs updating.

This rule has no "n/a." Every retroactive sweep claim, every "this is safe because…" assertion, every "no concerns" sign-off must be backed by a read in this session. The cost is 30 seconds of grep + read per claim; the benefit is not shipping false-positive fixes (and not missing real bugs that contradict your memory of how something works).

### Pre-merge checks

Before declaring a change done, run through these four questions. ~30 seconds per change, catches the classes of bug this codebase has actually shipped (B114 cross-tab overlay drift, B121 cross-feature state writes, the kind of stale-state-vs-new-code mismatches that come from incremental migrations). The right answer to any of these can be "n/a" — the point is to have considered it.

1. **What state does this write that other code reads?** List every var / ref / state your change writes. For each, grep one consumer and read it. Cross-feature state is where the bug class lives: B121 cleared `roomState.desc` and three other subsystems (Room panel, Lich Map findRoom, Genie Map memo, contacts effect) read it. The pre-check found the same-batch behavior was already absorbed by existing protections (B109 timer, F34 early-return); the after-the-fact discovery would have shipped as B12X. If your change writes nothing other code reads, the answer is "n/a" — say so.

2. **What cleanup runs on unmount vs. on hide?** Tabs in Lichborne don't unmount when you switch — they go `display:none` (pitfall #24). So `useEffect` cleanup only fires on real unmount (close session), not on hide. Anything that should re-fire on *show* (re-apply per-character settings, refresh layout-dependent measurements, reset transient state) needs an explicit activeId-watcher in App.tsx, NOT just a mount-effect in GameWindow. B114 shipped with this gap: theme-overlay re-apply only ran on theme change, not on tab switch, so two characters with different accessibility overlays bleed into each other. If your change relies on a mount/unmount lifecycle, ask "does this need to fire on show/hide too?"

3. **Does the current code handle state written by older versions of itself?** YAML profile files, localStorage entries, the genie parse cache, the panel layout state, custom themes — all of these were written by some prior Lichborne version a user already ran. When your change reads any of this data, what happens if the field is missing (older write), has a different shape (intermediate refactor), or has a value the new code doesn't expect? **Backward compatibility we care about runs in one direction only**: current code reading prior-version state. We do NOT care about older clients reading newer files — testers run the current build, not v0.8.4. The right answers: (a) treat missing fields as their type default (most optional YAML), (b) bump GENIE_CACHE_VERSION when the cached shape changes (pitfall #16), (c) write an explicit migration in `profile-migrations.ts` by SOURCE version when a field is renamed / restructured. The wrong answer is reading stale state and crashing / silently dropping data the user previously saved.

4. **Does this change the `CharacterProfile` or `SharedProfile` shape — and if so, where's the upgrade guardrail?** This is the mandatory profile-safety check; a user's whole setup lives in `profiles/*.yaml` + `_shared.yaml`, and corrupting it on upgrade is the worst bug class we can ship (pre-v0.6.4 wiped everyone's profiles, principle #8). **Always ask it, even when the answer is "no change."** Walk it explicitly:
   - **Did I add/rename/restructure a field on `CharacterProfile` or `SharedProfile` ([profile-types.ts](src/renderer/profile-types.ts)), or change what a `state.*` suffix means?** Adding a NEW *optional* field is non-breaking — it loads as `undefined` on old profiles and needs no migration (just a sensible default at the read site). Renaming, removing, retyping, or changing the meaning of an existing field IS breaking.
   - **For a breaking change**: bump the version constant (`SHARED_PROFILE_VERSION` / `CHARACTER_PROFILE_VERSION` in [profile.ts](src/renderer/profile.ts)) AND register a migration in [profile-migrations.ts](src/renderer/profile-migrations.ts) keyed by the SOURCE version. Migrations must be PURE (the file isn't rewritten unless every step succeeds). The forward guardrail already exists for free: `runMigrations` returns `null` for a future-version file so a downgrade preserves it on disk rather than clobbering it — don't defeat that.
   - **For a NEW sidecar file/format** (like the v0.10.0 `.lb.yaml` Transfer bundle, or the genie cache): give it its OWN `formatVersion` and reject newer-than-known on read (`parseProfileExport` rejects `formatVersion > current`; genie cache bumps `GENIE_CACHE_VERSION`). New folders (e.g. `Exports/`) need no migration but must be created on demand (`ensureExportsDir`).
   - **For a new per-character setting**: it's a new `state.*` suffix — non-breaking by construction (the dynamic `state:` map round-trips it), but decide whether it belongs in the Profile Transfer allowlist (`TRANSFER_CATEGORIES`, pitfall #56) and the `_shared.yaml` vs per-character split.
   The output is one line: "no profile-shape change" OR "added optional field X (default Y)" OR "breaking: bumped to vN + migration registered." Never leave it unstated — a silent schema drift is how a profile-eating bug ships.

These are not a substitute for testing — they're a substitute for the bug report you'd otherwise get next week. The right output is either "no concerns" + a one-line note in the commit message, or a tracked issue if a concern surfaces that's out of scope for the current change.

### Retroactive sweeps at version boundaries

When cutting a new version (or whenever there's a natural pause), do a focused read of the last 3–5 versions' Tracker entries with a different question: **not "did the change work" but "what could a tester still hit that hasn't been reported yet?"** The pattern that ships dormant bugs in this codebase:

- A new feature (Fxx) writes state or uses an effect lifecycle in a way that's correct for its own scope but interacts unexpectedly with an OLDER subsystem the author wasn't focused on.
- The bug is silent — no exception, no obvious wrong rendering. Just a corner case where state drifts (B114), focus jumps unexpectedly (F32 edge cases), or stats accumulate against the wrong target.
- It takes a specific user configuration to trigger: multi-character + accessibility overlay (B114), Ctrl+Tab from a modal text input (F32), Lich script with both `<streamWindow>` and `<pushStream>` for the same id (B123).

For each recent feature, ask the four pre-merge questions retroactively. Read the consumers of any state it writes. Verify the cleanup story on show/hide. Check whether older saved state still loads correctly. Don't refactor on speculation — log a real concern with a repro path, or call it clean. The retroactive sweep done for v0.8.7 found B114 (real, fixed in v0.8.7) and noted F32 edge cases (deferred, not fix-worthy without a tester report). That's the right output shape: distinguish "real bug worth fixing" from "future-feature concern" from "minor papercut without a clean fix" and don't conflate them.

**When to do this**: at the cut of every minor version (v0.x.0 → v0.x+1.0) at minimum, and ideally before each patch release when a meaningful feature shipped recently. The cost is one focused review session; the alternative is a tester finding it cold and the trust hit that comes with shipping obvious-in-retrospect bugs.

## Operational guardrails

- **Never `git commit` without explicit user approval.** Asking is mandatory.
- **Windows + PowerShell** is the host shell. Use PowerShell syntax (`$null`, `$env:VAR`). Bash available for POSIX scripts.
- `profiles/` is gitignored; never commit user data captured during testing.
- `passwords.json` lives in `app.getPath('userData')`, encrypted with Electron `safeStorage` (Windows DPAPI). Don't move it into the repo tree.
- Prefer fixing root causes over adding fallbacks.
- **When proposing a feature, check the principles first.** If it duplicates Lich functionality, say so and ask. If it lives in the display/configuration/profile space, propose freely.
- **Native menu & hotkeys — "never touch" rules (PERMANENT).** These protect a consistent, Windows-native experience; treat them as invariants, not redesign-phase rules:
  - **Never override an Electron-reserved chord with a Lichborne action.** The built-in **role** accelerators stay exactly as-is — don't rebind them and never assign any of them to a Lichborne feature: `Ctrl+C/V/X/Z/Y/A` (clipboard / undo / redo / select-all), `Ctrl+R` & `Ctrl+Shift+R` (reload), `Ctrl+0` / `Ctrl+±` (zoom), `Ctrl+M` (minimize), `Ctrl+W` (close), `Ctrl+Q` (quit), `F11` (fullscreen), `Ctrl+Shift+I` / `F12` (devtools). Overriding any of these makes Lichborne behave unlike every other Windows app.
  - **Never touch the pre-existing in-app Lichborne chords.** `Ctrl+Shift+Enter` (QuickSend), `Ctrl+Tab` (next character), `Ctrl+1–9` (jump to character N, 1-indexed — `Ctrl+1` = first char) are handled in [App.tsx](src/renderer/App.tsx) and must keep working unchanged.
- **Native menu — hotkey policy (PERMANENT).** `setupMenu()` in [main.ts](src/main/main.ts) is the full File·Edit·View·Tools·Lich·Window·Help tree (shipped v0.10.0; Window menu gained the Move-to-New/Main-Window decouple items in v0.11.0). Policy: **(a) do NOT assign keyboard accelerators to native menu items** — Lichborne items are **click-only**; the final hotkey set is still deferred until agreed (don't invent shortcuts). **(b)** Where a native menu item maps to one of the pre-existing in-app chords above, *display* it as a hint via `registerAccelerator: false` (so App.tsx stays the handler — no double-fire) **unless** the chord is Electron-reserved. Lichborne menu clicks route through the `menu-action` IPC → `lichborne:session-action` DOM-event bridge ([src/shared/menuActions.ts](src/shared/menuActions.ts)); `sendMenuAction` targets the **FOCUSED** window — the active GameWindow there handles session actions, App handles app-level ones.
